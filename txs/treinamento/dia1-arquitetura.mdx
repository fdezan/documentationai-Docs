## Treinamento: Tax Service — Motor de Calculo Externo (Sovos)
**Publico**: Consultores SAP MM/SD Pleno/Senior

---

## 1. O Que e o Tax Service (TXS)?

O TXS e o framework do SAP S/4HANA que conecta o **Pricing Engine** (Condition Technique) a um **motor de calculo de impostos externo** (Sovos, Vertex, Thomson Reuters).

**No S/4HANA Public Cloud**, o uso de motor externo e **obrigatorio** para calculo de impostos brasileiros — nao e mais possivel manter condition records de impostos manualmente como no On-Premise.

### Antes (On-Premise) vs Agora (Cloud)

| Aspecto | On-Premise | S/4HANA Cloud |
|---------|-----------|---------------|
| Tax Procedure | TAXBRA / TAXBRAJ | TAXBRAJ (adaptado) |
| Determinacao | Condition records manuais (FTXP, OBYZ) | Motor externo (Sovos) |
| Calculo | Formulas internas do pricing | HTTP call para Sovos |
| Manutencao de taxas | Consultor atualiza condition records | Equipe fiscal atualiza no Sovos |
| Compliance | Responsabilidade do cliente | Compartilhada com parceiro fiscal |

### Por Que Motor Externo?

- **Complexidade tributaria BR**: 9+ tipos de imposto (ICMS, IPI, PIS, COFINS, ISS, IRRF, CSLL, ICMS-ST, DIFAL)
- **Mudancas frequentes**: legislacao tributaria brasileira muda constantemente
- **Clean Core**: S/4HANA Cloud nao permite modificacoes no standard
- **Especializacao**: parceiros fiscais (Sovos) sao especialistas em compliance tributario

## 2. As 8 Fases do Fluxo TXS

Quando o SAP calcula precos (pricing) e encontra uma **condition type com Formula 279**, o TXS e acionado.

```
PRICING SAP (Formula 279)
    |
    v
[FASE 1] Entrada e Validacoes
    |  - Recebe KOMK/KOMP/KOMV do pricing
    |  - Enriquece com dados fiscais (LFD: CNPJ, IE, CFOP, NCM)
    |  - Valida se TXS esta ativo
    |  - Cria objeto de contexto
    v
[FASE 2] Decisao
    |  - E a condition de "service call" (TXSC)?
    |    SIM → vai para Fase 3 (HTTP call)
    |    NAO → vai para Fase 6 (mapeia do buffer)
    v
[FASE 3] Montagem do Payload (Request)
    |  - Monta JSON com: header, itens, localizacoes, parties
    |  - Inclui: NCM, CEST, CNPJ, IE, CNAE, CFOP, contribuinte
    |  - BAdI Released pode modificar o payload aqui
    v
[FASE 4] Comunicacao HTTP
    |  - Autentica via OAuth2 (Client Credentials)
    |  - Serializa payload para JSON (camelCase)
    |  - POST para o Sovos
    |  - Recebe resposta JSON
    v
[FASE 5] Processamento da Resposta
    |  - Desserializa JSON de volta para estrutura ABAP
    |  - Separa por tipo de imposto (ICMS, IPI, PIS, COFINS...)
    |  - Armazena no Response Buffer (cache)
    v
[FASE 6] Mapeamento para KOMV
    |  - Mapeia tax_type → condition types (ICMS → BX16/ICM1/BX10...)
    |  - Preenche KWERT (valor), KBETR (taxa), KAWRT (base)
    |  - Mapeia tax code externo → tax code S/4 (via tabela)
    v
[FASE 7] Verificacao
    |  - Pelo menos 1 condicao foi mapeada com sucesso?
    |  - Se nenhuma: erro "mapping_condition_error"
    v
[FASE 8] Pos-Processamento
    |  - Error handling (se houve erro)
    |  - BAdI post_processing
    |  - Limpa dados temporarios, retorna ao pricing
    v
PRICING SAP (KOMV preenchida com impostos)
```

### Conceito Fundamental: 1 HTTP Call, N Mapeamentos

O pricing procedure tem **varias condition types de imposto** (TXSC, BX16, ICM1, BIP0, BPI1, BCO1...), mas o TXS faz **apenas 1 chamada HTTP** ao Sovos:

- **TXSC** (Service Call): Faz o HTTP call e armazena a resposta no buffer
- **BX16, ICM1, ICM2...** (ICMS): Leem do buffer e mapeiam ICMS
- **BIP0, IPI1, IPI2...** (IPI): Leem do buffer e mapeiam IPI
- **BPI1, IPIS, IPSN...** (PIS): Leem do buffer e mapeiam PIS
- **BCO1, ICOF, ICON...** (COFINS): Leem do buffer e mapeiam COFINS

<Callout kind="info">
Cada imposto tem **multiplas** condition types. Mapeamento via tabela J_1BNFTXCOND.
</Callout>

## 3. Fluxo MM vs SD

### MM — Materials Management (Compras)

```
PO / GR / Invoice → Pricing → Formula 279 → TXS → Sovos → KOMV
```

| Aspecto | Valor |
|---------|-------|
| Application | TX |
| Payload | V1 (standard) ou DirectCalc |
| Parties | No header (1 fornecedor, 1 centro) |
| Cenarios | Purchase Order, Goods Receipt, Invoice Verification, STO |

### SD — Sales and Distribution (Vendas)

```
SO / Billing / Delivery → Pricing → Formula 279 → TXS → Sovos → KOMV
```

| Aspecto | Valor |
|---------|-------|
| Application | V |
| Payload | V2 (parties por item) |
| Parties | Por item (diferentes clientes/enderecos) |
| Cenarios | Sales Order, Billing, Delivery, Returns |

### Diferenca Principal: V1 vs V2

| Caracteristica | V1 (MM) | V2 (SD) |
|---------------|---------|---------|
| Party/Location | No header | Por item |
| Custom Fields YY1_* | Nao | Sim (extensions[]) |
| Tax Registration | Header level | Item level |

## 4. Glossario de Termos

| Termo | Significado |
|-------|-----------|
| **KOMK** | Estrutura de header do pricing (Communication Header) — contem BUKRS, KAPPL, KALSM, WAERK |
| **KOMP** | Estrutura de item do pricing (Communication Item) — contem MATNR, WERKS, MENGE, MWSKZ, STEUC |
| **KOMV** | Estrutura de condicao do pricing (Condition Value) — contem KSCHL, KBETR, KWERT, KAWRT, KINAK |
| **LFD** | Local Fiscal Data — dados fiscais brasileiros (CNPJ, IE, CFOP, NCM) enriquecidos na Fase 1 |
| **Formula 279** | Formula de pricing que aciona o TXS — atribuida nas condition types de imposto |
| **Condition Type** | Tipo de condicao no pricing procedure (ex: TXSC, BX16, BIP0, BPI1, BCO1) |
| **Tax Code (MWSKZ)** | Codigo de imposto S/4 (ex: I1, I2, J1) — mapeia para o tax code do motor externo |
| **Pricing Procedure** | Sequencia de calculo de precos (ex: TXBRA1) — define quais condition types participam |
| **Payload** | Estrutura JSON enviada ao Sovos com todos os dados para calculo fiscal |
| **Response Buffer** | Cache em memoria que armazena a resposta do Sovos para evitar chamadas HTTP duplicadas |
| **TXS_ACTIVATION** | Tabela de customizing que ativa/desativa o TXS por aplicacao e pricing procedure |
| **Clean Core** | Principio do S/4HANA Cloud: sem modificacoes no standard, apenas extensoes oficiais |
| **OAuth2** | Protocolo de autenticacao usado na comunicacao SAP → Sovos (Client Credentials Flow) |
| **BAdI** | Business Add-In — ponto de extensao SAP para adicionar logica customizada |

## 5. Condition Types do TXS Brasil

Cada imposto possui **multiplas** condition types. Mapeamento via tabela J_1BNFTXCOND.

| Imposto | Condition Types (KSCHL) | Fonte |
|---------|------------------------|-------|
| **Service Call** | TXSC | `if_txs_constants.abap:146` |
| **ICMS** | BX16, ICM1, ICM2, BX10, BX11, BX12, BIC1 | `if_logbr_ft_txs_constants.abap:93-99` |
| **IPI** | BIP0, IPI1, IPI2, BX20, BX21, BX22, BXPA, BIP1 | `:101-108` |
| **PIS** | BPI1, IPIS, IPSN, BX80, BX81, BX82, BPIP | `:110-116` |
| **COFINS** | BCO1, ICOF, ICON, BX70, BX71, BX72, BCOP | `:118-124` |
| **II** | BXIR, II01, BASB, BXIE, BXII | `:126-130` |
| **IRRF** | BW61, BX9U, BX9V | `if_txs_constants_br.abap:110-112` |
| **INSS** | BW71, BX9R, BX9S | `:113-115` |

<Callout kind="info">
ISS, CSLL, CBS: condition types nao confirmadas no codigo — validar no sistema SAP.
</Callout>

## 6. Checklist — O Que Voce Precisa Saber Antes de Configurar TXS

- [ ] Qual o **pricing procedure** do cenario? (ex: TXBRA1)
- [ ] O pricing procedure tem as **condition types** de imposto com Formula 279?
- [ ] A sequencia esta correta? (TXSC primeiro, depois as demais condition types de imposto)
- [ ] A tabela **TXS_ACTIVATION** esta configurada para a aplicacao (TX/V) e procedure?
- [ ] Ha excecoes em **TXS_ACTIVATION_EXCEPTION** para alguma empresa/centro?
- [ ] A **conexao HTTP** (Logical Port + OAuth2) esta configurada e testada?
- [ ] A tabela **TXS_TAX_CODE_MAP** tem os mapeamentos de tax code para todos os cenarios?
- [ ] Os **dados mestres** estao completos? (NCM, CNPJ, IE, CNAE, Material Origin, IBGE)
- [ ] O **motor Sovos** esta respondendo e configurado para os cenarios desejados?
- [ ] O **feature toggle** GLO_LOG_BR_EXT_JSON_RESP esta ativo para debug?
