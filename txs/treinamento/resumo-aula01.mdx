## O Que Foi Abordado Nesta Aula

Nesta primeira aula, entendemos **o que e** o motor de calculo externo (Tax Service / TXS), **por que** a SAP adotou esse modelo, e **como** funciona a comunicacao entre o SAP e o motor. Os temas foram:

- O principio de request/response aplicado ao calculo de impostos
- Por que a TAXBRA e a RVABRA **nao calculam** impostos (quem calcula e a formula 320 ou 279)
- Arquitetura do motor externo: 100% fora do SAP, sem codigo dentro do sistema
- O conceito de "Super IVA" — poucos IVAs que atendem todos os cenarios
- A plataforma Sovos: amparos proprios vs amparos Sovos, aprovacao de legislacoes
- Traducao tecnica de legislacao: condicao e consequencia
- Estrutura do payload (JSON): request e response
- Preco bruto vs preco liquido com motor
- Reforma tributaria: como o motor ja trata IBS e CBS
- Enriquecimento de payload: foi discutido como adicionar informacoes ao payload quando o standard nao atende

## Conceitos Fundamentais

### 1. Request e Response — O Principio do Motor

Toda operacao com motor de calculo segue o mesmo principio: o SAP envia uma **solicitacao** (request) contendo os dados do documento (material, fornecedor/cliente, valor, NCM, etc.), e o motor devolve uma **resposta** (response) com os impostos calculados.

**Quanto mais completo for o request, melhor sera o calculo.** Se faltam dados — por exemplo, o CNAE do fornecedor ou a indicacao de Simples Nacional — o motor pode calcular incorretamente. Na J1BTAX, o sistema encontra a primeira regra de excecao aplicavel com base nos dados disponiveis. Ja o motor e "gentil": ele **sempre devolve um calculo**, mesmo que impreciso. Cabe ao consultor garantir que os dados mestres estejam completos.

### 2. TAXBRA e RVABRA Nao Calculam Impostos

Essa e uma das afirmacoes mais importantes da aula. A TAXBRA e a RVABRA definem o **comportamento** de um imposto: se escritura, se contabiliza, se soma na base. Mas quem faz o calculo e:

- **Formula 320** — no modelo classico (CBT/J1BTAX)
- **Formula 279** — no modelo com motor externo (TXS)

A formula 279 e **standard SAP**. Ela faz a chamada API ao motor e mapeia o retorno para as condicoes da pricing. Nao e desenvolvimento dos parceiros (Sovos, Avalara, etc.).

### 3. Motor Externo — Nenhum Codigo Dentro do SAP

O motor de calculo nao e um addon. Nao existe **nenhuma linha de codigo** do motor dentro do SAP. A comunicacao acontece toda via API (HTTP/JSON). Isso significa que:

- A configuracao do lado SAP e **a mesma** independente do motor usado
- Trocar de motor (ex: de Thomson para Sovos) nao exige mudanca no SAP
- O que diferencia os motores sao as funcionalidades internas: flexibilidade, granularidade de regras, interface de usuario

**Por que a SAP externalizou?** Manter regras tributarias brasileiras (que mudam diariamente) dentro do ERP e extremamente custoso. A SAP preferiu manter a emissao de documentos eletronicos (DRC) e externalizar o calculo.

### 4. Super IVA — Um IVA Para Todos os Cenarios

No modelo classico, voce precisa de varios IVAs: um para credito, outro sem credito, um para frete, outro para ativo imobilizado. Com o motor, trabalha-se com **poucos IVAs** (conceito de "Super IVA"):

- O motor determina CFOP, CST e dedutibilidade — nao mais o IVA
- O campo **TAXSable** no payload de retorno define se o SAP usa ICM1 (toma credito) ou ICM2 (nao toma credito)
- Existem IVAs especificos para casos pontuais: frete (IVA 40), ativo imobilizado (com condicoes C1C/C2C)

**Na pratica**: nao precisa mais daquela pilha de IVAs. E tambem nao precisa da configuracao de determinacao de CFOP em SD — o motor determina.

### 5. Condicao e Consequencia — Traduzindo Legislacao

Toda legislacao tributaria pode ser decomposta em:
- **Condicao**: QUANDO acontece (UF origem, UF destino, NCM, XTIP, CEST, data, tipo de operacao)
- **Consequencia**: O QUE acontece (aliquota, base, suspensao, isencao)

**Exemplo real**: Decreto de MG sobre alcool etilico hidratado:
- Condicao: operacao interna em MG + NCM especifico + XTIP + CEST + a partir de 22/03/2023
- Consequencia: aliquota ICMS = 13,08% (ao inves de 18%)

Na J1BTAX, voce tem ~20 campos para criar regras de excecao. No motor, voce tem **~200 campos**. E possivel usar CEST, EAN, codigo de atividade, entre outros — algo impossivel na TAXBRA.

### 6. Amparos Sovos vs Amparos Proprios

Na plataforma Sovos existem dois tipos de regras:

| Tipo | Quem Cria | Quem Mantem | Exemplo |
|------|-----------|-------------|---------|
| **Amparo Sovos** | Time tributario da Sovos | Sovos (com aprovacao do cliente) | Aliquota de ICMS interestadual, pauta fiscal |
| **Amparo proprio** | Cliente/consultor | Cliente | Regime especial, credito presumido, liminar |

A Sovos monitora legislacoes e cria/atualiza amparos automaticamente. Porem, **nao e arbitraria** — toda atualizacao aparece como notificacao para o cliente aprovar ou recusar. O cliente pode discordar da interpretacao da Sovos e criar seu proprio amparo.

A conta global da Sovos tem ~13.900 amparos cadastrados, incluindo regras da reforma tributaria (IBS e CBS).

## O Que Muda na Pratica: CBT vs TXS

| Aspecto | CBT (TAXBRA/RVABRA) | TXS (Motor Externo) |
|---------|---------------------|---------------------|
| **Quem calcula** | Formula 320 + classes ABAP (BR Calc Tech) | Formula 279 + API externa |
| **Codigo no SAP** | Sim (formulas, classes, tabelas) | Nao (zero codigo do motor no SAP) |
| **CFOP** | Determinacao por configuracao SD/MM | Motor determina |
| **CST** | Direito fiscal no SAP | Motor determina |
| **Dedutibilidade** | IVA define se ICM1 ou ICM2 | Campo TAXSable no payload define |
| **IVAs necessarios** | Muitos (credito, sem credito, frete, ST...) | Poucos ("Super IVA") |
| **Preco bruto/liquido** | Complexo, formula 959, configuracao PIS/COFINS incluso | SD: trocar "N" por "G" no payload. MM: desenvolvimento |
| **Regras de excecao** | ~20 campos disponiveis | ~200 campos disponiveis |
| **Manutencao de regras** | Consultor atualiza J1BTAX | Motor/equipe fiscal mantem |
| **Reforma tributaria** | Depende de notas SAP | Motor ja tem as regras |
| **Extensibilidade** | Formula 920 (improviso) | BAdI antes + BAdI depois (standard) |
| **Debug** | J1BTAX, formula 320, ABAP debugger | Transacao TXS_TRACE (payloads JSON) + ABAP debugger |

## Duvidas Respondidas em Aula

### Rodrigo: "O treinamento serve so para Sovos ou para qualquer motor?"
**Resposta**: Serve para qualquer motor. A estrutura tecnica do lado SAP e a mesma para Sovos, Avalara, Thomson/IDT, Invoicy ou qualquer outro. O que muda sao as funcionalidades internas de cada motor.

### Fabio: "Como eu sei de onde o motor calculou?"
**Resposta**: Na plataforma Sovos, voce consegue entrar e ver qual legislacao foi aplicada. Peca acesso de consulta ao motor logo no inicio do projeto. Dentro do SAP, use a transacao TXS_TRACE para ver os payloads JSON (request e response).

### Washington: "Onde no SAP o motor devolve os calculos?"
**Resposta**: Dentro da formula 279, existe um metodo que pega o response do motor e faz o mapeamento para as condicoes da pricing. O Desan vai disponibilizar um catalogo tecnico com todos os pontos de debug.

### Washington: "E se o motor nao devolver direito fiscal e CFOP?"
**Resposta**: Isso pode acontecer quando se usa um conector proprio do motor ao inves do conector standard SAP. Quando se usa o conector SAP (Formula 279), o motor devolve CFOP, CST e todas as informacoes necessarias. Sempre valide que esta usando a integracao standard.

### Dani: "Morre o IVA?"
**Resposta**: Sim, na pratica sim. Voce trabalha com poucos IVAs (Super IVA). Nao precisa mais de um IVA para credito e outro sem credito — o motor determina isso pelo campo TAXSable.

### Fabio: "Como contabilizar em contas diferentes com Super IVA? Ex: CIAP de ativo imobilizado."
**Resposta**: Para ativo imobilizado existem condicoes especificas (C1C, C2C). Para cenarios custom, requer desenvolvimento via BAdI: zerar a condition ICM1 e atribuir o valor a uma ZCM1 com chave de conta diferente.

### Ana: "O motor atende Default e preco bruto/liquido?"
**Resposta**: Sim para ambos. Default funciona direto. Preco bruto/liquido: para SD e simples (mudar "N" para "G" no payload + condicao ICMI cuida do resto). Para MM precisa desenvolvimento.

### Ana: "O motor atende a questao do CFOP? Vem do SAP ou do motor?"
**Resposta**: Vem do motor. O SAP pega do retorno e coloca na Miro, na ordem de vendas, na fatura e na nota fiscal.

### Joao Saraiva (chat): "Como fica o direito fiscal?"
**Resposta**: Existe um unico direito fiscal chamado TXS. O SAP nao olha mais a configuracao do direito fiscal na J1BTAX — pega a CST diretamente do payload e atualiza na nota fiscal.

### Fabio: "Na versao On-Premise, a SAP nao soltou notas da reforma tributaria para TXS."
**Resposta**: Confirmado. Porem o desenvolvimento necessario e simples: ler o payload de retorno (que ja vem com IBS/CBS calculados) e colocar nas conditions e na nota fiscal. Funciona como um "Control+C, Control+V" do payload para as estruturas SAP.

### Luiz: "E se o cliente tiver um regime especial alem da regra da Sovos?"
**Resposta**: Crie um amparo proprio. Exemplo: se alem da reducao geral de ICMS o cliente tem aliquota de 10% por regime especial, cadastre uma regra com CNPJ + UF + NCM + XTIP + CEST e a aliquota especifica.

### Wilson: "O TAXSable e documentado pela SAP?"
**Resposta**: Nao tem documentacao explicita. Foi descoberto via debug no codigo. E o tipo de conhecimento que so se adquire debugando.

### Wilson: "Ja encontrou imposto que o motor nao atendeu?"
**Resposta**: Sim — produtor rural (FunRural, Senar). O motor calcula, mas o SAP nao tem integracao standard para receber. Recentemente a SAP lancou uma tabela extensivel que deve resolver isso em breve.

### Angela: "O motor integra com TM (Transportation Management)?"
**Resposta**: Se integrar TM com MM usando mestre de servicos, nao integra (SAP so chama o motor quando tem dado mestre de material). Se integrar TM com SD, integra normalmente.

### Participante: "Tem muito problema de de-para de campos entre SAP, CPI e motor?"
**Resposta**: Sim, e uma confusao enorme. Cada sistema usa nomenclatura diferente para o mesmo campo. Existe um mapa de campos que ajuda.

## Dicas Praticas para Consultores

### Antes de Comecar o Projeto

1. **Peca acesso de consulta ao motor** (Sovos, Avalara, etc.) logo no inicio. Voce vai precisar para entender por que o motor calculou ou nao calculou como esperado.

2. **Garanta dados mestres completos**: CNAE, natureza legal, regime tributario (Simples Nacional ou nao), contribuinte de ICMS ou nao — esses 4 campos sao **obrigatorios** para o motor funcionar corretamente.

3. **Importe as Best Practices** mas **nao ative o Scope 43D de uma vez**. Revise configuracao por configuracao no Solution Builder para entender o que e relevante para seu cenario.

### Durante o Projeto

4. **Use a transacao TXS_TRACE** para monitorar os payloads JSON. Ali voce ve exatamente o que o SAP enviou (request) e o que o motor devolveu (response).

5. **O motor e chamado uma unica vez por acao** na pricing (numa condition especifica). As demais condicoes leem do buffer. Se mudar quantidade ou algum dado, recalcula.

6. **Cuidado ao criar regras no motor**: testar em cenarios completos. No exemplo da aula, uma regra para zerar ICMS em compras de consumo funcionou, mas gerou efeito colateral — o motor parou de calcular Default.

7. **Tres formas de customizar**: (a) regra no motor, (b) enriquecimento via CPI (fora do SAP), (c) BAdI ABAP (dentro do SAP). No Public Cloud, a opcao (c) pode nao estar disponivel para todas as BAdIs.

8. **MM e mais dificil que SD** quando se trata de preco bruto/liquido. Em SD e literalmente trocar um campo. Em MM precisa desenvolvimento para liquidar a pricing.

### Mudanca de Mindset

9. **Voce vai precisar entender legislacao.** Com motor de calculo, o consultor e forcado a olhar para a legislacao de forma que a J1BTAX nunca exigiu. Isso e positivo — seu conhecimento tributario vai aumentar consideravelmente.

10. **Nao dependa so do motor.** Mesmo com Sovos mantendo ~13.900 amparos, o consultor precisa validar se o calculo faz sentido para o cenario especifico do cliente. Divergencias de interpretacao sao comuns.

## Reforma Tributaria e o Motor

Para quem trabalha com motor de calculo, a reforma tributaria (IBS e CBS) foi a melhor noticia possivel:

- **Public Cloud**: praticamente transparente. O motor ja tinha as regras. Bastou emitir com data a partir de 01/01/2026. Alguns clientes nem contrataram consultoria.

- **On-Premise**: a SAP demorou a lancar notas. O desenvolvimento necessario e simples — ler os valores de IBS/CBS do payload de retorno e colocar nas conditions e na nota fiscal. O instrutor chama de "desenvolvimento Control+C, Control+V".

- **Motor Sovos**: ja tem amparos cadastrados para CBS, IBS, ClessTrib, incluindo cenarios como suspensao para atividade naval, venda de ativo imobilizado com mais de 12 meses, reducao de 60% para produtos de higiene.

## Glossario Rapido

| Termo | Definicao |
|-------|-----------|
| **TXS** | Tax Service — framework do SAP que conecta a pricing a um motor de calculo externo |
| **Formula 279** | Formula standard da pricing que faz a chamada API ao motor. Substitui a formula 320 |
| **Formula 320** | Formula classica que faz o calculo via J1BTAX/TAXBRA |
| **Payload** | Estrutura JSON padrao da SAP contendo os dados para calculo (request) e o resultado (response) |
| **TAXSable** | Campo no response que indica se o imposto e dedutivel (credito) ou nao |
| **taxiCode** | Campo no response que contem o CFOP determinado pelo motor |
| **Super IVA** | Conceito de usar poucos IVAs com motor — o comportamento e definido pelo motor, nao pelo IVA |
| **Amparo Sovos** | Regra criada/mantida pela Sovos com base em legislacao publica |
| **Amparo proprio** | Regra criada pelo cliente para regime especial ou excecao |
| **CPI** | Cloud Platform Integration — middleware no BTP por onde o payload pode transitar |
| **Scope 43D** | Escopo de Best Practices da SAP para ativacao do TXS |
| **TXS_TRACE** | Transacao SAP para monitorar payloads JSON (request/response) |
| **ICMI** | Condicao de preco em SD que viabiliza preco bruto/liquido |
| **ICM1 / ICM2** | Condicoes de ICMS: ICM1 = com credito, ICM2 = sem credito |
| **Condicao e Consequencia** | Logica de traducao tecnica de legislacao: QUANDO x acontecer, APLIQUE y |
| **Convencio** | Regra tributaria federal que precisa de decreto estadual para entrar em vigor |
| **Decreto** | Instrumento do governador que valida um convencio no ambito estadual |
| **Enriquecimento** | Adicionar campos ou informacoes ao payload que o standard nao envia |
| **BAdI antes / depois** | Pontos de extensao na formula 279: antes de enviar ao motor e depois de receber o retorno |
