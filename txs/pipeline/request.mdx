## Visao Geral do Pipeline

O TXS (Tax Service) opera como um pipeline de 8 fases, acionado pela formula 279 dentro da pricing procedure do S/4HANA. O fluxo completo e:

| Fase | Descricao | Classes Principais |
|------|-----------|-------------------|
| **1** | **Entrada e Validacoes** | CL_LOGBR_FILL_LFD_ON_PRINCING, CL_TXS_SWITCH, CL_TXS_PRICING_CONTEXT |
| **2** | **Decisao: Service Call vs Mapping** | CL_TXS_CONTROLLER, CL_TXS_PAYLOAD_TYPE |
| **3** | **Build do Payload** | CL_TXS_PAYLOAD_BR, CL_TXS_ITEMS_BR, CL_TXS_LOCATIONS_BR, CL_TXS_PARTY_BR |
| 4 | Comunicacao HTTP | CL_TXS_COMMUNICATION, CL_TXS_CONNECTION |
| 5 | Processamento da Response | CL_TXS_RESPONSE, CL_TXS_TAX_TYPES_VALUES_BR |
| 6 | Mapeamento para KOMV | CL_TXS_CONDITIONS_MAPPER_BR, CL_TXS_COND_MAPPER_FACTORY |
| 7 | Verificacao de Mapeamento | CL_TXS_CONDITION_PROCESSING |
| 8 | Pos-processamento | CL_TXS_CONTROLLER, CL_TXS_LFD_RECORDER |

**Este documento cobre as fases 1, 2 e 3** -- tudo que acontece desde a entrada no pricing ate o payload JSON estar pronto para a chamada HTTP ao motor externo de calculo.

**Constantes criticas**:
- Condition de service call: **TXSC** (Fonte: `if_txs_constants.abap:146` -- `SERVICE_CALL type KSCHA value 'TXSC'`)
- Formula de pricing: **279** (Fonte: `cl_txs_condition_processing.abap:71` -- `con_formula_279 type KOFRM value 0000279`)
- Inativacao de condition: KINAK = **'Z'** (Fonte: `if_txs_constants.abap:83`)

## Fase 1 -- Entrada e Validacoes

### 1.1 Ponto de Entrada: Formula 279

O TXS e acionado pela **formula 279** na pricing procedure. Quando o pricing engine processa uma condition type que possui a formula 279 atribuida (KOFRM = 0000279), ele chama:

```
CL_TXS_CONTROLLER::IF_TXS_CONTROLLER~EXECUTE
```

Este metodo recebe como parametros todas as estruturas do pricing:

| Parametro | Tipo | Descricao |
|-----------|------|-----------|
| `i_pricing_header` | `KOMK` | Cabecalho do pricing (empresa, moeda, parceiro, application) |
| `c_pricing_item` | `KOMP` (changing) | Item do pricing (material, quantidade, planta, tax code) |
| `i_pricing_condition_records` | `TXS_KOMV_INDEX_TTY` | Todas as conditions do pricing |
| `i_plants_or_branches` | `T001W` | Dados da planta/filial |
| `i_countries` | `T005` | Dados do pais |
| `i_transaction_code` | `SYST_TCODE` | Transacao corrente (ME21N, VA01, MIRO, etc.) |
| `i_local_time` | `TIMS` | Hora local do sistema |
| `i_pricing_processing_moment` | `CHAR1` | Momento do processamento |
| `i_group_condition` | `GKOMV` | Condicao de grupo |
| `c_pricing_condition_record` | `KOMV_INDEX` (changing) | Condition record corrente |
| `c_pricing_condition_value` | `KWERT` (changing) | Valor da condition |
| `c_sy_subrc` | `SYST_SUBRC` (changing) | Codigo de retorno |

Fonte: `cl_txs_controller.abap:402-561`

**Mecanismo**: A formula 279 e chamada uma vez para cada condition type na pricing procedure que a utiliza. A primeira chamada e para a condition **TXSC** (service call), que dispara a chamada HTTP ao Tax Engine. As chamadas subsequentes sao para as conditions de cada imposto (ex: BX16, ICM1, BIP0, etc.), que fazem leitura do buffer de response.

---

### 1.2 Preenchimento do LFD (Local Fiscal Data)

**Primeira acao** do metodo `EXECUTE` -- antes de qualquer validacao:

```abap
DATA(fill_pricing_with_lfd) = NEW cl_logbr_fill_lfd_on_princing( ).
c_pricing_item = fill_pricing_with_lfd->run(
  i_pricing_header = i_pricing_header
  i_pricing_item   = c_pricing_item
).
```

Fonte: `cl_txs_controller.abap:404-408`

A classe `CL_LOGBR_FILL_LFD_ON_PRINCING` enriquece o item de pricing (KOMP) com dados fiscais brasileiros. Campos preenchidos:

| Campo KOMP | Descricao | Origem |
|------------|-----------|--------|
| `STEUC` | NCM (classificacao fiscal do material) | `MARA-STEUC` / `MARC-STEUC` |
| `TXJCD` | Tax Jurisdiction Code (codigo IBGE do municipio) | Endereco da planta/parceiro |
| `J_1BMATORG` (MTORG) | Material Origin (origem do material) | `EKPO-J_1BMATORG` |
| `J_1BMATUSE` (MTUSE) | Material Usage (uso do material) | `EKPO-J_1BMATUSE` |

**Importancia**: Se o LFD nao preencher corretamente estes campos, todo o restante do pipeline sera afetado -- NCM vazio impede classificacao fiscal, IBGE vazio impede determinacao de ISS, material usage errado afeta calculo de ICMS.

**Debug**: Breakpoint em `CL_LOGBR_FILL_LFD_ON_PRINCING::RUN`. Verificar `c_pricing_item` apos retorno.

### 1.3 BAdI Pre-Processing

Imediatamente apos o LFD, o controller tenta executar o BAdI `TXS_BADI_CONTROLLER::PRE_PROCESSING`:

```abap
GET BADI badi_controller.
CALL BADI badi_controller->pre_processing
  EXPORTING
    i_pricing_header            = i_pricing_header
    i_pricing_condition_records = i_pricing_condition_records
    i_plants_or_branches        = i_plants_or_branches
    i_transaction_code          = i_transaction_code
    ...
  CHANGING
    c_pricing_item              = c_pricing_item
    c_pricing_condition_record  = c_pricing_condition_record
    c_pricing_condition_value   = c_pricing_condition_value
    c_sy_subrc                  = c_sy_subrc
  RECEIVING
    r_overwrite_standard        = lv_overwrite_standard.
```

Fonte: `cl_txs_controller.abap:413-434`

**Comportamento critico**: Se `R_OVERWRITE_STANDARD = ABAP_TRUE`, o metodo `EXECUTE` faz `RETURN` imediatamente -- **toda a logica TXS e pulada**.

Fonte: `cl_txs_controller.abap:436-438`

**Status**: Este BAdI **NAO e Released** (`TXS_BADI_CONTROLLER`). Interface: `IF_EX_TXS_BADI_CONTROLLER`.
Fonte: `incoming/if_ex_txs_badi_controller.abap:7-24`

**Casos de uso**: Desativar TXS em cenarios especificos, enriquecer KOMP antes do processamento, desviar para logica customizada.

### 1.4 Verificacao de Ativacao

Apos o pre-processing (se nao houve overwrite), o controller verifica se o TXS esta ativo. A verificacao segue uma sequencia de tres niveis:

```abap
IF cl_txs_switch=>get_instance( )->is_active(
     i_application       = i_pricing_header-kappl
     i_pricing_procedure = i_pricing_header-kalsm
     i_country           = i_countries-land1
     i_company_code      = i_pricing_header-bukrs ) = abap_false.
  IF lr_controller->is_service_call_cond_type( c_pricing_condition_record-kschl ).
    c_pricing_condition_record-kinak = if_txs_constants=>txsc_condition_inactive.  " 'Z'
  ENDIF.
  RETURN.
ENDIF.
```

Fonte: `cl_txs_controller.abap:469-477`

#### Sequencia de Verificacao em CL_TXS_SWITCH

**Nivel 1 -- Conexao Ativa** (`CONSTRUCTOR`):

```abap
data(lr_connection_config) = cl_txs_conn_config_factory=>get_instance( )->get_connection_config( ).
if lr_connection_config is bound.
  active = lr_connection_config->is_connection_active( ).
endif.
```

Fonte: `cl_txs_switch.abap:76-80`

Verifica se existe uma configuracao de conexao ativa para o Tax Engine (Logical Port / Communication Arrangement). Se nao houver conexao configurada, `active = abap_false` e o TXS nao processa.

**Nivel 2 -- Tabela TXS_ACTIVATION** (`IS_INTEGRATION_ACTIVE`):

```abap
r_active = cl_txs_activation_dao=>get_instance( )->get_tax_service_active_status(
  i_application  = i_application
  i_pricing_name = i_pricing_procedure ).
```

Fonte: `cl_txs_switch.abap:113-115`

Verifica se a combinacao `application` (V=SD, TX=MM) + `pricing_procedure` (ex: TXBRA1) esta registrada na tabela de ativacao.

**Nivel 3 -- Tabela TXS_ACTIV_EXCEPT (excecoes)** (`CHECK_ACTIVATION_EXCEPTIONS`):

```abap
data(ls_activation_exception) = lr_activation_dao->get_tax_service_except_entry(
  i_application  = i_application
  i_pricing_name = i_pricing_procedure
  i_country      = i_country
  i_company_code = i_company_code ).

r_active = xsdbool( ls_activation_exception is initial or
   ls_activation_exception-calculation_engine = if_txs_constants=>calculation_engine-partner_engine ).
```

Fonte: `cl_txs_switch.abap:54-70`

Se existir uma excecao, o TXS so continua ativo se `calculation_engine = 'P'` (Partner Engine).
Constante: `if_txs_constants=>calculation_engine-partner_engine` = `'P'` (Fonte: `if_txs_constants.abap:200`)

**Nivel 4 -- Activation Control V2** (fallback):

Se o nivel 2 retornar `false`, ha um fallback adicional:

```abap
if r_active = abap_false and ( cl_txs_activation_control=>get_instance(
    )->if_txs_activation_control~is_integration_active(
        i_country = i_country  i_company_code = i_company_code ) = abap_true ).
  r_active = abap_true.
endif.
```

Fonte: `cl_txs_switch.abap:124-128`

#### Resultado

Se o TXS estiver inativo (`is_active = false`):
- A condition TXSC recebe `KINAK = 'Z'` (inativa)
- O metodo `EXECUTE` retorna sem processar

#### Validacoes Adicionais Apos Ativacao

Apos a verificacao de ativacao, o controller ainda verifica:

1. **Importacao de mercadorias** (Brasil): Se for cenario de PO de importacao, retorna sem processar.
   Fonte: `cl_txs_controller.abap:479-485`

2. **Isencao de imposto**: Se o header indicar transacao isenta (`CL_TXS_TAX_EXEMPTION`), retorna.
   Fonte: `cl_txs_controller.abap:488-490`

3. **Modo display**: Se `KOMK-TRTYP` indica modo de exibicao, nao processa.
   Fonte: `cl_txs_controller.abap:496`

## Fase 2 -- Decisao: Service Call vs Mapping

### 2.1 PROCESS_CONDITION_RECORD

O metodo `PROCESS_CONDITION_RECORD` e o ponto de bifurcacao do pipeline. Ele decide se a condition atual deve disparar uma chamada HTTP (service call) ou ler do buffer de response (mapping):

```abap
method process_condition_record.
  data(lr_payload_type) = cl_txs_payload_type=>get_instance( pricing_context->get_country( ) ).
  data(lv_payload_type) = lr_payload_type->determine_payload_type( pricing_context ).

  " Material vazio = nao processa (exceto Canada e V2)
  if ( pricing_context->get_material_number( ) is initial and
       pricing_context->get_country( ) <> if_txs_constants=>country_key-canada )
    and lv_payload_type = if_txs_payload_type=>con_payload_types-payload.
    return.
  endif.

  if is_service_call_cond_type( c_pricing_condition_record-kschl ).
    call_txs_from_pricing_formula( ).    " TXSC -> HTTP call
  else.
    map_txs_to_pricing_formula( ... ).   " Outras conditions -> buffer read
  endif.

  cl_txs_condition_processing=>get_instance( country )->check_mapping( ... ).
endmethod.
```

Fonte: `cl_txs_controller.abap:647-669`

**Logica de decisao**:

```
condition_record-kschl e TXSC (service call)?
  |
  +-- SIM -> call_txs_from_pricing_formula()
  |           -> initialize()
  |           -> build_payload()        <-- Fase 3
  |           -> calculate_taxes()      <-- Fase 4 (HTTP)
  |
  +-- NAO -> map_txs_to_pricing_formula()
              -> Le response do buffer (ja calculado)
              -> Mapeia tax_type para condition value
```

A verificacao de service call usa internal codes:

```abap
method is_service_call_cond_type.
  r_is_service_call_cond_type = cl_txs_internal_codes=>get_instance( )->is_value_in_internal_code(
    value         = i_cond_type
    component     = pricing_context->get_component( )
    country       = pricing_context->get_country( )
    internal_code = cl_txs_internal_codes=>condition_type-tax_service_integration ).
endmethod.
```

Fonte: `cl_txs_controller.abap:612-617`

O internal code `CONDITION_TYPE_TAX_SERVICE_INTEGRATION` mapeia para **TXSC** na configuracao.
Fonte: `cl_txs_internal_codes.abap:29`

### 2.2 Determinacao do Tipo de Payload

A classe `CL_TXS_PAYLOAD_TYPE` (com country specialization `CL_TXS_PAYLOAD_TYPE_BR`) determina qual tipo de payload sera construido:

| Payload | Classe Builder | Constante | Quando |
|---------|---------------|-----------|--------|
| **V1** (padrao) | `CL_TXS_PAYLOAD` / `CL_TXS_PAYLOAD_BR` | `'CL_TXS_PAYLOAD'` | Cenario padrao BR (MM e SD sem V2) |
| V2 | `CL_TXS_REQUEST_V2` | `'CL_TXS_REQUEST_V2'` | SD com integracao V2 ativa |
| Direct Calc | `CL_TXS_DIRECT_CALC_PAYLOAD` | `'CL_TXS_DIRECT_CALC_PAYLOAD'` | MM com calculo direto |

Fonte: `if_txs_payload_type.abap:7-9`

**Brasil sempre usa V1**: A classe `CL_TXS_PAYLOAD_TYPE_BR` sobrescreve a determinacao e, para cenarios padrao brasileiros, retorna o payload V1. O V2 e usado apenas quando explicitamente ativado para SD.

Constante de versao V2: `PAYLOAD_VERSION_V2` = `'V2'` (Fonte: `if_txs_constants.abap:204`)

## Fase 3 -- Build do Payload

### 3.1 Sequencia de Build

O build e disparado por `call_txs_from_pricing_formula`:

```abap
method call_txs_from_pricing_formula.
  initialize( ).       " Limpa payload, response, communication
  build_payload( ).    " Constroi o payload JSON
  calculate_taxes( ).  " Envia HTTP (Fase 4)
endmethod.
```

Fonte: `cl_txs_controller.abap:217-222`

O metodo `build_payload` usa a factory para obter o builder correto e executar o build:

```abap
method build_payload.
  payload = cl_txs_payload_factory=>get_instance( )->get_payload( pricing_context ).
  payload->set_pricing_context( pricing_context ).
  payload->build( ).
endmethod.
```

Fonte: `cl_txs_controller.abap:169-173`

A factory `CL_TXS_PAYLOAD_FACTORY` usa o pattern **Country Specialization**: tenta instanciar `CL_TXS_PAYLOAD_BR` e, se nao existir, faz fallback para `CL_TXS_PAYLOAD`.
Fonte: `cl_txs_payload_factory.abap:51-79`

O metodo `IF_TXS_PAYLOAD~BUILD` em `CL_TXS_PAYLOAD` executa na seguinte ordem:

```
1. INITIALIZE          -> CLEAR payload (limpa todos os campos)
2. BUILD_HEADER        -> Popula campos escalares do header
                          BR: chama super->build_header() + adiciona operation_nature_code
3. BUILD_ITEM          -> Constroi tabela _items via CL_TXS_ITEMS=>GET_INSTANCE
4. BUILD_LOCATIONS     -> Constroi tabela _locations via CL_TXS_LOCATIONS=>GET_INSTANCE
5. BUILD_PARTY         -> Constroi tabela _party via CL_TXS_PARTY=>GET_INSTANCE
6. BUILD_BP_EXEMPTION  -> Constroi tabela _bp_exemption_details via CL_TXS_BP_EXEMPTIONS=>GET_INSTANCE
7. APPLY_CUSTOM_LOGIC  -> Executa BAdI BD_GSLOG_EXT_TAX_CALC_PAYLOAD::EXTEND_PAYLOAD
                          (pode modificar QUALQUER campo do payload, incluindo header e items)
```

Fonte: `cl_txs_payload.abap:219-227`

### 3.2 Header (TXS_PAYLOAD)

A estrutura raiz `TXS_PAYLOAD` e construida por `CL_TXS_PAYLOAD_BR::BUILD_HEADER`.

#### Campos-Chave do Header

| Campo | Builder | Fonte SAP | Descricao |
|-------|---------|-----------|-----------|
| `id` | `CL_TXS_DAO::GENERATE_PAYLOAD_ID` | UUID (`CL_SYSTEM_UUID`) | Identificador unico da requisicao |
| `company_id` | `CL_TXS_PAYLOAD::BUILD_HEADER` | `KOMK-BUKRS` | Codigo da empresa |
| `currency` | `CL_TXS_PAYLOAD::BUILD_HEADER` | `KOMK-WAERK` | Moeda ISO 4217 |
| `date` | `CL_TXS_PAYLOAD_BR::GET_PAYLOAD_DATE` | Varia por cenario | Data da transacao (ISO 8601) |
| `saleor_purchase` | `CL_TXS_PAYLOAD::GET_SALES_OR_PURCHASE` | `KOMK-KAPPL` | `'sale'` (V) ou `'purchase'` (TX) |
| `gross_or_net` | `CL_TXS_PAYLOAD_BR::GET_GROSS_OR_NET` | Logica BR | `'gross'` ou `'net'` |
| `operation_nature_code` | `CL_TXS_PAYLOAD_BR::GET_OPERATION_NATURE_CODE` | Derivado | Natureza da operacao (P, S, PI, SI, STO...) |

#### Logica de `date` (BR)

Fonte: `cl_txs_payload_br.abap:257-325`

```
1. Application = SD (KAPPL = 'V')?
   +-- Billing doc transfer? -> KOMK-FKDAT (billing date)
   +-- Stock transfer (STO)?  -> KOMK-FKDAT se preenchido, senao KOMK-ERDAT
   +-- Outros SD              -> KOMK-PRSDT (pricing date)

2. STO via MM (nao SD)?       -> KOMK-PRSDT (pricing date)

3. Default (outros MM)        -> SY-DATUM + SY-TIMLO (data/hora do sistema)
```

**Conversao de timezone**: Converte para o timezone da planta via `CL_TXS_FM_DATE_TIME=>GET_INSTANCE( )->GET_LOCATION_TIMEZONE` usando pais, regiao e CEP da planta. Se nao encontrar, usa `SY-ZONLO`.

#### Logica de `gross_or_net` (BR)

Fonte: `cl_txs_payload_br.abap:76-98`

```
1. E downpayment / accounts receivable / customer down payment?
   +-- SIM -> 'gross'

2. Gross price ativo E application = TX (MM) E NAO e invoice?
   +-- SIM -> 'gross'

3. Default -> 'net'
```

Constantes: `cl_txs_internal_codes=>api_gross_or_net-gross` = `'API_GROSS'`, `-net` = `'API_NET'`
Fonte: `cl_txs_internal_codes.abap:205-206`

#### Logica de `operation_nature_code` (BR)

Fonte: `cl_txs_payload_br.abap:101-119`

A determinacao e delegada para `CL_TXS_OPERATION_NATURE_BR` (Singleton), com roteamento:

```
is_stock_transfer()?
  +-- SIM -> GET_OPERATION_NATURE_CODE_STO
  +-- NAO ->
        KAPPL = 'V' (SD) -> GET_OPERATION_NATURE_CODE_SD
        KAPPL = 'TX' (MM) -> GET_OPERATION_NATURE_CODE_MM
```

**Valores de Operation Nature** (Fonte: `if_txs_constants_br.abap:122-148`):

| Valor | Constante | Descricao |
|-------|-----------|-----------|
| `P` | `purchase` | Compra padrao |
| `PI` | `purchase_invoice` | Invoice de compra (MIRO) |
| `S` | `sale` | Venda padrao |
| `SI` | `sale_invoice` | Fatura de venda (billing doc transfer) |
| `STO` | `stock_transfer_outgoing` | Transferencia estoque -- saida |
| `STI` | `stock_transfer_incoming` | Transferencia estoque -- entrada |
| `CI` | `consignment_invoice` | Invoice de consignacao |
| `CS` | `consignment_shipping` | Remessa de consignacao |
| `CM` | `credit_memo_req` | Nota de credito |
| `DM` | `debit_memo_req` | Nota de debito |
| `RO` | `return_with_own_nf` | Devolucao com NF propria |
| `RC` | `return_with_customer_nf` | Devolucao com NF do cliente |
| `RV` | `returns_to_vendor` | Devolucao ao fornecedor |
| `GI` | `subcontr_goods_issue` | Saida para subcontratacao (mvt 541) |
| `GR` | `subcontr_goods_receipt` | Recebimento subcontratacao (mvt 542) |
| `TPS` | `third_party_shipment` | Remessa de terceiro |
| `TPI` | `third_party_invoice` | Fatura de terceiro |

### 3.3 Items (TXS_ITEM + Sub-Estruturas)

O build de items e feito por `CL_TXS_ITEMS_BR` (herda de `CL_TXS_ITEMS`), instanciada via country specialization.

Fluxo de montagem (Fonte: `cl_txs_items.abap:84-91`):

```
CL_TXS_ITEMS::ADD_ITEM
  -> build_header()              --> campos escalares do item
  -> build_item_classifications() --> CL_TXS_ITEM_CLASSIFICATIONS_BR
  -> build_additional_item_info() --> CL_TXS_ADDITIONAL_ITEM_INFO_BR
  -> build_cost_information()     --> CL_TXS_COST_INFORMATION_BR
  -> build_exemption_detail()     --> CL_TXS_MAT_EXEMPTIONS
```

#### Campos Escalares do Item

| Campo | Builder | Fonte SAP | Descricao |
|-------|---------|-----------|-----------|
| `id` | `GET_ID` | `KOMP-KPOSN` | Identificador sequencial do item |
| `item_code` | `GET_ITEM_CODE` | `KOMP-MATNR` | Numero do material |
| `item_type` | `GET_ITEM_TYPE` | `is_service()` | `'material'` ou `'service'` |
| `quantity` | `GET_QUANTITY` (BR) | Logica BR | Quantidade tributavel |
| `unit_price` | `GET_UNIT_PRICE` (BR) | Logica MM/SD | Preco unitario |
| `tax_code` | (base) | `KOMP-MWSKZ` | Codigo de imposto |
| `shipping_cost` | `GET_SHIPPING_COST` (BR) | MM freight only | Flag frete incoming |

#### Logica unit_price

**MM** (Fonte: `cl_txs_items_br.abap:107-145`):
1. Determina condition type via internal code `CONDITION_TYPE_BASE_AMOUNT`
2. Busca KWERT no pricing para essa condition type
3. `unit_price = KWERT / quantidade`
4. Fallback: `NETPR / KPEIN` (net price da classe base)

**SD** (Fonte: `cl_txs_items_br.abap:148-166`):
1. Determina condition type via internal code `TAAS_UNIT_PRICE_BASE` (Fonte: `if_txs_constants_br.abap:45`)
2. Busca KWERT no pricing para essa condition type
3. `unit_price = KWERT / quantidade`
4. Fallback: `NETPR / KPEIN`

Formatacao: 12 casas decimais, separador ponto, sem notacao cientifica.

#### Item Classifications (NCM, CEST, EAN)

Builder: `CL_TXS_ITEM_CLASSIFICATIONS_BR` (Fonte: `cl_txs_item_classifications_br.abap:106-116`)

Despacho: STO -> `APPEND_ITEM_CLASS_FOR_STO`, Servico -> `APPEND_ITEM_CLASS_FOR_SERVICE`, Material -> `APPEND_ITEM_CLASS_FOR_MATERIAL`

| System Code | Metodo | Fonte do Valor | Descricao |
|-------------|--------|----------------|-----------|
| `ncm` | `GET_MATERIAL_NCM` | `KOMP-STEUC` -> `format_ncm()` | NCM 8 digitos sem pontos |
| `cest` | `GET_MATERIAL_CEST` | `dao_br->get_cest(STEUC, MATNR)` | CEST do material |
| `ean` | `GET_MATERIAL_EAN` | `KOMP-EAN11` | EAN-13/GTIN |
| `exTipi` | `GET_MATERIAL_EXTIPI` | `KOMP-STEUC` -> `get_extipi()` | Excecao TIPI (ultimos 2-3 digitos NCM) |

Para **servicos**, as classificacoes sao: `servTypeExec` (tipo servico municipal no local de execucao), `servTypeFed` (codigo ABRASF federal), `servTypeFrom` (tipo servico no local do fornecedor/planta), `NBS` (Nomenclatura Brasileira de Servicos).

#### Additional Item Information (Origem, Uso, CBTA)

Builder: `CL_TXS_ADDITIONAL_ITEM_INFO_BR` (Fonte: `cl_txs_additional_item_info_br.abap:217-233`)

| Type | Descricao | Fonte SAP |
|------|-----------|-----------|
| `materialorigin` | Origem do material (0-8) | `KOMP-MTORG` (`EKPO-J_1BMATORG`) |
| `usage` | Uso do material | MM: `KOMP-MTUSE`, SD: via tax_code_sd + `CL_TXS_USAGE_BR` |
| `usagepurpose` | Proposito de uso | MM: `MTUSE + tax_code`, SD: `doc_type + item_cat + tax_code_sd` |
| `ownproduction` | Flag producao propria (SD only) | `KOMP-OWNPR` |
| `taxclassification` | Classificacao fiscal (BR01-BR09) | `dao_br->select_mat_tax_classification(material, date)` |
| `subcontracting` | Info subcontratacao | `pricing_context->is_subcontracting()` + movement type |
| `cprb` | Relevancia CPRB | `CL_TXS_CPRB_BR` (SD e servicos) |
| `referenceid` | GUID do TXS_TRACE | `CL_TXS_REFERENCE_ID_BR` |
| `commercialquantity` | Quantidade comercial | Quando unidade tributavel difere da comercial |
| `laborcontract` | Contrato trabalho (REINF) | Servicos: `dao_br->get_reinf_service_type()` (MM) ou tax_code (SD) |

**CBTA** (Complementary Business Transaction Attributes): Quando a feature `complementary_business_attr` esta ativa, os campos `business_transaction`, `material_usage` e `usage_purpose` sao lidos de `KOMP-txs_business_transaction` / `KOMP-txs_material_usage` / `KOMP-txs_usage_purpose` e validados contra grupos de internal codes.

#### Cost Information (Desconto, Frete, Seguro)

Builder: `CL_TXS_COST_INFORMATION_BR` (Fonte: `cl_txs_cost_information_br.abap:49-80`)

**Regra importante**: STO **nao popula** cost_information (retorna vazio).

| Cost Type | Internal Code | Fonte do Valor |
|-----------|---------------|----------------|
| `discount` | `TAAS_TOTAL_DISCOUNT` | `ABS(KWERT)` da condition type mapeada |
| `laborContractDiscount` | `CONDITION_TYPE_LABOR_CONTRACTS_INSS_BASE_DISCOUNT` | Iteracao nas conditions |
| `freight` | `TAAS_TOTAL_FREIGHT` | `ABS(KWERT)` da condition type mapeada |
| `insurance` | `TAAS_TOTAL_INSURANCE` | `ABS(KWERT)` da condition type mapeada |
| `otherExpenses` | `TAAS_TOTAL_OTHER_EXPENSE` | `ABS(KWERT)` da condition type mapeada |

Fonte dos internal codes: `if_txs_constants_br.abap:41-44`

A resolucao do condition type usa `dao->select_cond_type_by_int_code()`, que consulta a tabela `J_1BTXCOND` para obter o KSCHL real do pricing procedure ativo.

### 3.6 Extensoes

#### BAdI BD_GSLOG_EXT_TAX_CALC_PAYLOAD (Released)

**Status**: Released -- seguro para S/4HANA Public Cloud
**Fase TXS**: 3 (ultimo passo do build, antes da chamada HTTP)
**Metodo**: `EXTEND_PAYLOAD`

Chamado em `CL_TXS_PAYLOAD::APPLY_CUSTOM_LOGIC` (Fonte: `cl_txs_payload.abap:68-92`):

```abap
GET BADI ext_tax_calc_payload.
CALL BADI ext_tax_calc_payload->extend_payload
  EXPORTING
    pricing_header      = pricing_header      " KOMK
    pricing_item        = pricing_item        " KOMP
    lfd_for_consumption = lfd_for_consumption " IF_GSLOG_LFD_FOR_CONSUMPTION
  CHANGING
    payload             = payload.            " TXS_PAYLOAD (TUDO)
```

**O que pode modificar**: QUALQUER campo do payload -- header, items, locations, party, bp_exemption, extensions.

**Campos header nao populados no standard** (so via BAdI): `cash_discount_percent`, `business_partner_id`, `is_transaction_in_tax_group`, `is_comp_deferred_tax_enabled`, `is_cash_discount_planned`.

#### Custom Fields YY1_*

A classe `CL_TXS_CUSTOM_FIELD_EXTRACTOR` detecta automaticamente campos com prefixo `YY1_` nas estruturas KOMK/KOMP via RTTI e os inclui em `_extensions[]` (header) e `item_extensions[]` (item).

Fonte: `cl_txs_custom_field_extractor.abap:33-40`

**Como funciona**: Campos customizados criados via Fiori App Custom Fields (F1481) recebem prefixo `YY1_` automaticamente. O extractor filtra por `name+0(4) = 'YY1_'` e monta a tabela `TXS_EXTENDED_FIELDS` com nome e valor de cada campo.

#### Estrutura TXS_EXTENDED_FIELDS

```
{ custom_field: "YY1_SPECIAL_TAX_FLAG", custom_field_value: "X" }
```

Existe em dois niveis: `TXS_PAYLOAD-_extensions` (header) e `TXS_ITEM-item_extensions` (item).
Fonte: `txs_extended_fields.abap:1-10`

## Checklist do Consultor -- Antes do HTTP

Verificacoes a fazer no payload antes que ele seja enviado ao Tax Engine:

### Header

- [ ] `operation_nature_code` correto para o cenario (P/S/PI/SI/STO/STI/CM/DM/GI/GR...)
- [ ] `date` no formato ISO 8601 (`YYYY-MM-DDTHH:MM:SS.000Z`) e no timezone correto da planta
- [ ] `saleor_purchase` coerente com application (sale=SD, purchase=MM)
- [ ] `gross_or_net` correto (downpayment=gross, MM com gross price ativo e nao-invoice=gross, demais=net)
- [ ] `currency` preenchido (ISO 4217)
- [ ] `company_id` preenchido (BUKRS)

### Items

- [ ] `unit_price` > 0 (preco unitario nao pode ser zero)
- [ ] `quantity` > 0 (quantidade nao pode ser zero)
- [ ] `item_code` preenchido (numero do material)
- [ ] NCM preenchido (8 digitos, sem pontos) em `item_classifications`
- [ ] Material origin preenchido em `additional_item_information` (se material importado)
- [ ] Material usage coerente com cenario (0=Revenda, 1=Industrializacao, 2=Consumo, 3=Ativo)
- [ ] Cost information com frete/seguro/desconto quando aplicavel (exceto STO)

### Locations

- [ ] SHIP_FROM e SHIP_TO presentes (para material e STO)
- [ ] CONTRACT_FROM e CONTRACT_TO presentes (para servico e frete)
- [ ] `state` (UF) preenchido em todas as locations (essencial para ICMS)
- [ ] `country` = `'BR'` em todas as locations domesticas
- [ ] `zip_code` (CEP) preenchido
- [ ] Inversao correta em subcontratacao e consignment return

### Parties

- [ ] CNPJ preenchido (14 digitos, sem formatacao) em ambas as parties
- [ ] IE (Inscricao Estadual) preenchida (STCD3 ou STATE_INSC)
- [ ] CNAE preenchido (codigo CNAE da atividade economica)
- [ ] Status de contribuinte correto para todos os impostos (ICMSContributor, IPIContributor, etc.)
- [ ] `taxDeclarationType` preenchido (LP=Lucro Presumido, LR=Lucro Real, SN=Simples Nacional)
- [ ] SUFRAMA presente quando operacao em Zona Franca
- [ ] `isNaturalPerson` correto (afeta LGPD e log de payload)
- [ ] Codigo IBGE preenchido (IBGECode)

### Extensoes

- [ ] Campos YY1_* sendo extraidos corretamente (se customizados)
- [ ] BAdI `BD_GSLOG_EXT_TAX_CALC_PAYLOAD` nao corrompendo dados standard
