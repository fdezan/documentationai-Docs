## Visao Geral

Apos todas as conditions serem mapeadas (Fase 6 -- Mapeamento para KOMV), duas fases finais executam no pipeline TXS:

- **Fase 7 -- Verificacao de Mapeamento**: Verifica se pelo menos uma condition type foi mapeada com sucesso. Se nenhuma mapeou, gera excecao `mapping_condition_error`.
- **Fase 8 -- Pos-Processamento**: Grava dados fiscais no LFD (`GSLOG_LOCAL_FISCAL_DATA`), executa BAdI `post_processing`, e trata erros com gravacao de application log (SLG1).

Estas fases sao criticas porque:
1. A Fase 7 protege contra cenarios de pricing silenciosamente vazio (todas as conditions excluidas sem erro)
2. A Fase 8 conecta o resultado do TXS ao fluxo fiscal brasileiro (LFD para Nota Fiscal) e garante tratamento robusto de excecoes

## Fase 7 -- Verificacao de Mapeamento

### 7.1 CL_TXS_CONDITION_PROCESSING::CHECK_MAPPING

O metodo `CHECK_MAPPING` e chamado ao final de `CL_TXS_CONTROLLER::PROCESS_CONDITION_RECORD`, apos cada condition ser processada (seja por service call HTTP ou por leitura do buffer/mapeamento).

Fonte: `cl_txs_controller.abap:666-668`:
```abap
cl_txs_condition_processing=>get_instance( country )->check_mapping(
  condition = c_pricing_condition_record
  procedure = pricing_context->get_pricing_procedure( ) ).
```

**Responsabilidade**: Rastrear se pelo menos uma condition type foi mapeada com sucesso dentro da pricing procedure. Se a ultima condition for atingida e nenhuma teve sucesso, gerar excecao.

**Instanciacao**: Singleton via `GET_INSTANCE( country )`, com country specialization (padrao SAP: tenta `CL_TXS_CONDITION_PROCESSING_{country}`, fallback para `CL_TXS_CONDITION_PROCESSING`).
Fonte: `cl_txs_condition_processing.abap:57-66`

### 7.2 Logica de Verificacao

O metodo `CHECK_MAPPING` implementa a seguinte logica:

Fonte: `cl_txs_condition_processing.abap:99-127`:
```abap
method if_txs_condition_processing~check_mapping.
  if is_service_call_condition( condition ).
    clear mapped.
    clear last_condition.
    return.
  endif.

  if mapped = abap_true.
    return.
  endif.

  if condition-kinak = if_txs_constants=>condition_inactivity_indicator-used_internally.
    if is_last_condition( condition ).
      raise exception type cx_txs_exception
        exporting
          textid      = if_txs_const_exceptions=>mapping_condition_error
          attrib1     = conv #( procedure )
          attrib2     = conv scx_attrname( condition-kappl )
          msg_display = if_txs_const_exceptions=>message_type_default
          msg_type    = if_txs_const_exceptions=>message_type_default.
    endif.
  else.
    mapped = abap_true.
  endif.
endmethod.
```

**Passo a passo da logica**:

1. **Se e a condition de service call (TXSC)**: Reseta o flag `mapped` e limpa `last_condition`. Isso reinicia a verificacao para cada novo item/chamada.
   - Verifica via `CL_TXS_INTERNAL_CODES` se o `KSCHL` da condition esta configurado como `condition_type-tax_service_integration`
   Fonte: `cl_txs_condition_processing.abap:146-151`

2. **Se `mapped = abap_true`**: Pelo menos uma condition ja mapeou com sucesso -- retorna imediatamente (OK).

3. **Se `condition-kinak = 'Z'` (used_internally)**: A condition foi excluida internamente pelo mapper (nao corresponde a nenhum tax type retornado pelo Tax Engine).
   Fonte: `if_txs_constants.abap:13` -- `used_internally type kinak value 'Z'`
   - **Se e a ultima condition (formula 279)**: NENHUMA condition mapeou -- gera excecao `mapping_condition_error`
   - **Se NAO e a ultima**: Continua verificando as proximas conditions

4. **Se `condition-kinak` e diferente de `'Z'`** (normalmente vazio/space): A condition esta ativa e mapeou com sucesso -- seta `mapped = abap_true`.

**Resumo decisorio**:

| KINAK da Condition | E a Ultima? | mapped | Resultado |
|--------------------|-------------|--------|-----------|
| `' '` (space/vazio) | N/A | Seta `true` | OK -- condition mapeou |
| `'Z'` (used_internally) | Nao | Mantem `false` | Continua verificando |
| `'Z'` (used_internally) | Sim | `false` | ERRO: `mapping_condition_error` |
| `'Z'` (used_internally) | Sim | `true` | Nao chega aqui (retorna no passo 2) |
| TXSC (service call) | N/A | Reseta para `false` | Reinicia verificacao |

### 7.3 Formula 279

A formula 279 e o marcador que identifica quais conditions sao gerenciadas pelo TXS na pricing procedure. Todas as condition types TXS (TXSC e as conditions de imposto) devem ter esta formula atribuida.

Fonte: `cl_txs_condition_processing.abap:71`:
```abap
constants con_formula_279 type kofrm value 0000279.
```

O metodo `DETERMINE_LAST_CONDITION` percorre todas as conditions da pricing procedure e identifica a **ultima** que possui formula 279:

Fonte: `cl_txs_condition_processing.abap:69-81`:
```abap
method determine_last_condition.
  constants con_formula_279 type kofrm value 0000279.
  data condition type komv_index.

  loop at conditions assigning field-symbol(<line>) where kofrm = con_formula_279.
    condition = <line>.
  endloop.

  last_condition = condition.
endmethod.
```

**Importante**: O LOOP percorre **todas** as conditions com formula 279 e a variavel `condition` recebe o valor da **ultima** entrada (comportamento do LOOP sem EXIT). Ou seja, a ultima condition com formula 279 na tabela `conditions` e considerada a "ultima condition TXS".

O metodo `IS_LAST_CONDITION` compara a condition atual com a `last_condition` determinada:

Fonte: `cl_txs_condition_processing.abap:135-143`:
```abap
method is_last_condition.
  if last_condition is initial.
    determine_last_condition( ).
  endif.

  if last_condition-kschl = condition-kschl and last_condition-kposn = condition-kposn.
    result = abap_true.
  endif.
endmethod.
```

Compara tanto o `KSCHL` (condition type) quanto o `KPOSN` (posicao do item) para garantir identificacao correta.

As conditions sao carregadas no inicio do processamento via `SET_CONDITIONS`:

Fonte: `cl_txs_condition_processing.abap:130-132`:
```abap
method if_txs_condition_processing~set_conditions.
  conditions = pricing_conditions.
endmethod.
```

Chamado em `IF_TXS_CONTROLLER~EXECUTE`:
Fonte: `cl_txs_controller.abap:449`:
```abap
cl_txs_condition_processing=>get_instance( i_countries-intca )->set_conditions( i_pricing_condition_records ).
```

### 7.4 Erro mapping_condition_error

Quando a excecao e gerada:

- **Classe de excecao**: `CX_TXS_EXCEPTION`
- **TextID**: `if_txs_const_exceptions=>mapping_condition_error`
- **Message class**: `TXS_MESSAGES`, message number `083`
  Fonte: `if_txs_const_exceptions.abap:370-377`:
  ```abap
  constants:
    begin of mapping_condition_error,
      msgid type symsgid value 'TXS_MESSAGES',
      msgno type symsgno value '083',
      attr1 type scx_attrname value 'ATTRIB1',
      attr2 type scx_attrname value 'ATTRIB2',
      attr3 type scx_attrname value '',
      attr4 type scx_attrname value '',
    end of mapping_condition_error.
  ```

**Atributos da excecao**:
- `attrib1` = nome da pricing procedure (ex: `TXBRA1`)
- `attrib2` = application (`V` para SD, `TX` para MM)

**Causas comuns**:
1. Tabela `J_1BNFTXCOND` nao configurada para os condition types usados na pricing procedure
2. O Tax Engine nao retornou nenhum tax type na response (ex: cenario fiscal nao suportado)
3. Todos os condition types da pricing procedure foram excluidos (KINAK = 'Z') porque nenhum tax type retornado corresponde a nenhuma condition
4. Conditions de imposto na pricing procedure sem formula 279 atribuida

## Fase 8 -- Pos-Processamento

A Fase 8 engloba tres atividades executadas apos o processamento das conditions:
1. Gravacao de dados fiscais no LFD (especifico Brasil)
2. BAdI `post_processing` (extensibilidade)
3. Tratamento de erros (error handling)

### 8.1 Record LFD Taxes (Especifico Brasil)

O registro de dados fiscais no LFD (`GSLOG_LOCAL_FISCAL_DATA`) e uma responsabilidade da classe brasileira `CL_TXS_CONTROLLER_BR`, que herda de `CL_TXS_CONTROLLER` e redefine o metodo `CALL_TXS_FROM_PRICING_FORMULA`.

Fonte: `cl_txs_controller_br.abap:58-62`:
```abap
method call_txs_from_pricing_formula.
  call method super->call_txs_from_pricing_formula( ).
  record_lfd_taxes( ).
endmethod.
```

**Sequencia**:
1. `super->call_txs_from_pricing_formula()` -- executa o fluxo standard: `initialize()` -> `build_payload()` -> `calculate_taxes()` -> `clear_context()`
   Fonte: `cl_txs_controller.abap:217-222`
2. `record_lfd_taxes()` -- grava os valores fiscais calculados no LFD

**Metodo RECORD_LFD_TAXES**:

Fonte: `cl_txs_controller_br.abap:114-123`:
```abap
METHOD record_lfd_taxes.
  DATA(lv_item_number) = cl_txs_lfd_key_determiner=>get_instance(
    )->get_lfd_key_item_number( pricing_context ).
  DATA(lv_document_number) = cl_txs_lfd_key_determiner=>get_instance(
    )->get_lfd_key_document_number( pricing_context ).
  DATA(lt_tax_value) = cl_txs_tax_values_br=>get_instance(
    )->get_tax_values( pricing_context ).

  cl_txs_lfd_recorder=>get_instance( )->fill_lfd_taxes(
    item_number     = lv_item_number
    document_number = lv_document_number
    tax_values      = lt_tax_value ).
ENDMETHOD.
```

**Classes envolvidas**:

| Classe | Responsabilidade |
|--------|------------------|
| `CL_TXS_LFD_KEY_DETERMINER` | Determina o numero de item e documento para gravacao no LFD. Despacha para subclasses por cenario. |
| `CL_TXS_TAX_VALUES_BR` | Extrai os valores fiscais calculados pelo TXS a partir do pricing context. |
| `CL_TXS_LFD_RECORDER` | Grava os valores fiscais na tabela `GSLOG_LOCAL_FISCAL_DATA`. |

**Subclasses do Key Determiner** (por cenario de negocio):

| Subclasse | Cenario |
|-----------|---------|
| `CL_TXS_LFD_KEY_DETERMINER_PO` | Purchase Order (ME21N/ME22N) |
| `CL_TXS_LFD_KEY_DETERMINER_MIGO` | Goods Receipt (MIGO) |
| `CL_TXS_LFD_KEY_DETERMINER_MIRO` | Invoice Verification (MIRO) |
| `CL_TXS_LFD_KEY_DETERMINER_SD` | Sales & Distribution (VA01/VF01) |

**Nota**: O `record_lfd_taxes()` e chamado **apenas** quando a condition processada e a TXSC (service call), pois esta dentro do `call_txs_from_pricing_formula`. Para as demais conditions (que fazem buffer read via `map_txs_to_pricing_formula`), o LFD ja foi gravado na passagem da TXSC.

### 8.2 Add Response (Especifico Brasil)

A classe `CL_TXS_CONTROLLER_BR` tambem redefine o metodo `ADD_RESPONSE` para armazenar a resposta do Tax Engine em formato consumivel por outros componentes de localizacao:

Fonte: `cl_txs_controller_br.abap:33-55`:
```abap
METHOD add_response.
  DATA(response_consuption) = logbr_external_tax_response=>get_instance( ).
  DATA(taxes_response) = response->get_response( ).
  DATA response_line TYPE if_logbr_external_tax_response=>logbr_ext_response.

  super->add_response( ).

  MOVE-CORRESPONDING taxes_response TO response_line-response EXPANDING NESTED TABLES.

  IF line_exists( taxes_response-tax_lines[ 1 ] ).
    response_line-document_item = COND #(
      WHEN cl_txs_business_transaction=>get_instance( )->is_invoice( )
        THEN pricing_context->get_item( )-taxps
      ELSE taxes_response-tax_lines[ 1 ]-id ).
  ENDIF.

  TRY.
      response_line-document_reference = pricing_context->get_item_reference( ).
    CATCH cx_txs_invalid_reference.
  ENDTRY.

  response_consuption->set_response( CORRESPONDING #( response_line ) ).
ENDMETHOD.
```

**Finalidade**: Disponibiliza a response do Tax Engine via `LOGBR_EXTERNAL_TAX_RESPONSE` para consumo por outros componentes da localizacao brasileira (ex: Nota Fiscal). Diferencia entre invoice (usa `taxps` do item) e outros cenarios (usa `id` da primeira tax line).

### 8.3 BAdI Post-Processing

Apos todo o processamento (incluindo o `CATCH` de excecoes), o BAdI `TXS_BADI_CONTROLLER::POST_PROCESSING` e chamado.

Fonte: `cl_txs_controller.abap:538-557`:
```abap
TRY.
    GET BADI badi_controller.
    CALL BADI badi_controller->post_processing
      EXPORTING
        i_pricing_header            = i_pricing_header
        i_pricing_condition_records = i_pricing_condition_records
        i_plants_or_branches        = i_plants_or_branches
        i_transaction_code          = i_transaction_code
        i_local_time                = i_local_time
        i_price_determination       = i_price_determination
        i_pricing_processing_moment = i_pricing_processing_moment
        i_countries                 = i_countries
        i_group_condition           = i_group_condition
      CHANGING
        c_pricing_item              = c_pricing_item
        c_pricing_condition_record  = c_pricing_condition_record
        c_pricing_condition_value   = c_pricing_condition_value
        c_sy_subrc                  = c_sy_subrc.
  CATCH cx_badi_not_implemented ##NO_HANDLER.
ENDTRY.
```

**Caracteristicas**:

| Aspecto | Detalhe |
|---------|---------|
| **BAdI** | `TXS_BADI_CONTROLLER` |
| **Metodo** | `POST_PROCESSING` |
| **Interface** | `IF_EX_TXS_BADI_CONTROLLER` |
| **Status** | **NAO Released** -- risco em upgrades |
| **Pode sobrescrever standard?** | Nao (nao retorna `R_OVERWRITE_STANDARD`) |

Fonte da assinatura: `incoming/if_ex_txs_badi_controller.abap:25-40`

**Parametros CHANGING**:
- `c_pricing_item` (KOMP) -- pode alterar dados do item de pricing
- `c_pricing_condition_record` (KOMV_INDEX) -- pode alterar a condition record (valor, aliquota, base)
- `c_pricing_condition_value` (KWERT) -- pode alterar o valor da condition
- `c_sy_subrc` (SYST_SUBRC) -- pode alterar o codigo de retorno

**Importante**: O `post_processing` e chamado **apos** o bloco TRY/CATCH de excecoes. Ou seja, ele executa mesmo se houve erro no processamento (o erro ja foi tratado no CATCH). Isso significa que o BAdI pode:
- Ajustar valores de conditions apos o mapeamento
- Gravar logs customizados
- Corrigir valores que o error handler setou (ex: restaurar prsok)

**Quando NAO e chamado**:
- Se o BAdI `pre_processing` retornou `r_overwrite_standard = abap_true` (toda a logica TXS foi pulada, incluindo o post_processing)
  Fonte: `cl_txs_controller.abap:436-438`

### 8.4 Error Handling

Quando qualquer excecao `CX_TXS_EXCEPTION` e lancada durante o processamento das conditions, o bloco CATCH no metodo `IF_TXS_CONTROLLER~EXECUTE` executa o tratamento de erros.

Fonte: `cl_txs_controller.abap:505-534`:
```abap
CATCH cx_txs_exception INTO DATA(lx_txs_exception).

  TRY.
      GET BADI badi_controller.
      CALL BADI badi_controller->error_handling
        EXPORTING
          i_exception                = lx_txs_exception
          i_pricing_condition_record = c_pricing_condition_record
          i_pricing_item             = c_pricing_item
        RECEIVING
          r_overwrite_standard       = lv_overwrite_standard.
    CATCH cx_badi_not_implemented.
      CLEAR lv_overwrite_standard.
  ENDTRY.

  IF lv_overwrite_standard = abap_false.

    IF lr_controller->is_service_call_cond_type( c_pricing_condition_record-kschl ).
      lr_controller->clear_resp_buffer_by_error( ).
    ENDIF.

    lr_controller->add_app_log( i_exception = lx_txs_exception ).

    cl_txs_scen_det_param_context=>get_instance( )->clear_context( ).

    lr_controller->handle_pricing_error(
      EXPORTING i_exception                = lx_txs_exception
      CHANGING  c_pricing_item             = c_pricing_item
                c_pricing_condition_record = c_pricing_condition_record ).
  ENDIF.
```

**Sequencia de tratamento** (4 passos):

**Passo 1 -- BAdI error_handling** (pode suprimir o erro):
- BAdI `TXS_BADI_CONTROLLER::ERROR_HANDLING` e chamado primeiro
- Se retornar `r_overwrite_standard = abap_true`: todo o tratamento standard e pulado (o BAdI assumiu o tratamento)
- Se `cx_badi_not_implemented`: nenhuma implementacao ativa, `lv_overwrite_standard = abap_false` (tratamento standard executa)

Fonte da assinatura: `incoming/if_ex_txs_badi_controller.abap:41-47`:
```abap
methods ERROR_HANDLING
  importing
    !I_EXCEPTION                type ref to CX_TXS_EXCEPTION
    !I_PRICING_ITEM             type KOMP
    !I_PRICING_CONDITION_RECORD type KOMV_INDEX
  returning
    value(R_OVERWRITE_STANDARD) type ABAP_BOOL.
```

**Passo 2 -- clear_resp_buffer_by_error()** (limpa buffer contaminado):
- Executado apenas se a condition atual e do tipo service call (TXSC)
- Remove a response cacheada no buffer para o item que deu erro, evitando que respostas parciais/incorretas sejam reutilizadas por conditions subsequentes

Fonte: `cl_txs_controller.abap:233-265`:
```abap
method clear_resp_buffer_by_error.
  data(lr_response_buffer) = cl_txs_response_buffer=>get_instance( ).
  data(ls_item) = pricing_context->get_item( ).
  data(ls_header) = pricing_context->get_header( ).
  ...
  if ls_item-matnr is not initial or cl_txs_payload->is_aut_txcode_sel_possible( sy-tcode ).
    if pricing_context->is_downpaymnt_and_calc_buf_atv( ).
      lr_response_buffer->if_txs_response_buffer2~clear_by_item_id(
        cl_txs_calculation_data_buffer=>get_instance( )->get_document_item_id( ) ).
    else.
      lr_response_buffer->if_txs_response_buffer2~clear_by_item_id( ls_item-kposn ).
    endif.
  else.
    lr_response_buffer->if_txs_response_buffer2~clear_by_tax_details(
      i_s4_tax_code           = ls_item-mwskz
      i_tax_jurisdiction_code = ls_header-txjcd
      i_tax_date              = ls_header-prsdt ).
  endif.
endmethod.
```

A limpeza varia conforme o cenario:
- Se material preenchido ou auto tax code selection possivel: limpa por `item_id` (kposn)
- Caso contrario: limpa por `tax_details` (tax code + jurisdiction + date)
- Para downpayment com buffer ativo: usa `document_item_id` do calculation data buffer

**Passo 3 -- add_app_log()** (grava no SLG1):
- Grava a excecao completa no Application Log

Fonte: `cl_txs_controller.abap:120-133`:
```abap

  data lv_response_error_json type string.

  cl_txs_log_dao=>get_instance( )->add_exception_to_log(

    i_payload   = lv_payload_json
```

**Nota sobre pessoa fisica**: Se o payload e de pessoa fisica (`is_payload_from_natural_person() = abap_true`), os JSONs de payload e response **nao sao gravados** no log (protecao LGPD). Apenas a excecao e logada.

## Cenarios de Problema Comuns

### Cenario 1: mapping_condition_error

**Sintoma**: Erro na pricing com mensagem "mapping condition error" (TXS_MESSAGES 083). Pricing marcado como incorreto.

**Causa raiz**: Nenhuma condition type na pricing procedure foi mapeada com sucesso. Todas receberam `KINAK = 'Z'` (exclusao interna).

**Por que acontece**:
1. Tabela `J_1BNFTXCOND` nao configurada para os condition types usados (ex: BX16, ICM1 para ICMS; BIP0, IPI1 para IPI; etc.)
2. Tax Engine nao retornou nenhum tax type aplicavel ao cenario
3. Os condition types na pricing procedure nao correspondem aos tax types retornados
4. Formula 279 nao atribuida a todas as condition types TXS

**Debug**:
1. Breakpoint em `CL_TXS_CONDITION_PROCESSING::CHECK_MAPPING`
2. Verificar `mapped` -- se sempre `false` ate a ultima condition
3. Verificar `condition-kinak` de cada condition -- se todas `'Z'`
4. Verificar response do Tax Engine (Debug Point 15) -- quais tax types retornou

**Solucao**: Configurar `J_1BNFTXCOND` com mapeamento correto entre condition types da pricing procedure e tax types do Tax Engine.

### Cenario 2: prsok = false (Pricing Incorreto)

**Sintoma**: Documento nao pode ser criado ou salvo. Pricing marcado como incorreto (`prsok = false`). Condition TXSC com `KINAK = 'X'`.

**Causa raiz**: Qualquer excecao `CX_TXS_EXCEPTION` em qualquer fase do pipeline TXS desencadeia o error handler, que marca o pricing como incorreto.

**Por que acontece**:
- Erro HTTP (timeout, Tax Engine offline)
- Tax code invalido (retornado pelo engine mas nao mapeado no S/4)
- Erro de configuracao (TXS_ACTIVATION, J_1BNFTXCOND)
- Erro de mapeamento (mapping_condition_error)
- Qualquer outra excecao TXS

**Debug**:
1. Breakpoint em `CL_TXS_CONTROLLER::HANDLE_PRICING_ERROR`
2. Inspecionar `i_exception` -- message class, message number, atributos
3. Verificar SLG1 (objeto `TXS_LOG`) para detalhes completos com payload e response JSON

**Solucao**: A solucao depende da causa raiz. Consultar SLG1 para identificar a excecao especifica e tratar conforme o cenario (ver `reference/txs-troubleshooting.md` para os 10 cenarios documentados).

### Cenario 3: LFD Vazio (Dados Fiscais Nao Gravados)

**Sintoma**: Nota Fiscal criada sem dados de imposto. Tabela `GSLOG_LOCAL_FISCAL_DATA` vazia para o documento.

**Causa raiz**: O metodo `RECORD_LFD_TAXES` nao foi executado ou falhou silenciosamente.

**Por que acontece**:
1. `CL_TXS_CONTROLLER_BR` nao esta sendo instanciado (country specialization nao ativa)
2. A condition processada nao e TXSC (LFD so e gravado na passagem do service call)
3. `CL_TXS_LFD_KEY_DETERMINER` retornou chave incorreta (item/documento number)
4. `CL_TXS_TAX_VALUES_BR::GET_TAX_VALUES` retornou tabela vazia
5. Excecao ocorreu antes de `record_lfd_taxes()` (no `super->call_txs_from_pricing_formula()`)

**Debug**:
1. Breakpoint em `CL_TXS_CONTROLLER_BR::RECORD_LFD_TAXES`
   - Se o breakpoint nao e atingido: verificar se `CL_TXS_CONTROLLER_BR` esta sendo instanciado
   - Verificar `CL_TXS_CONTROLLER::CREATE_INSTANCE` -- deve criar `CL_TXS_CONTROLLER_BR` para pais BR
2. Inspecionar `lv_item_number` e `lv_document_number` -- devem estar preenchidos
3. Inspecionar `lt_tax_value` -- deve conter os valores fiscais calculados
4. Verificar se `cl_txs_lfd_recorder=>get_instance( )->fill_lfd_taxes()` executa sem erro

**Solucao**: Verificar a instanciacao country-specific e garantir que o TXS esta calculando impostos corretamente antes de investigar o LFD.

### Cenario 4: BAdI Sobrescrevendo Resultado

**Sintoma**: Valores de imposto nas conditions alterados inesperadamente apos o processamento TXS. Valores diferem do retornado pelo Tax Engine.

**Causa raiz**: Implementacao ativa do BAdI `TXS_BADI_CONTROLLER::POST_PROCESSING` modificando valores de KOMV.

**Por que acontece**:
1. Implementacao do BAdI altera `c_pricing_condition_record-kwert` (valor)
2. Implementacao do BAdI altera `c_pricing_condition_record-kbetr` (aliquota)
3. Implementacao do BAdI altera `c_pricing_condition_value`
4. Implementacao do BAdI altera `c_pricing_item` (dados do item)

**Debug**:
1. Breakpoint no `post_processing` do BAdI: `CL_TXS_CONTROLLER::IF_TXS_CONTROLLER~EXECUTE` (linhas 538-557)
2. Verificar se ha implementacoes ativas do BAdI `TXS_BADI_CONTROLLER`
3. Comparar valores de `c_pricing_condition_record` antes e depois da chamada do BAdI

**Solucao**: Revisar implementacoes ativas do BAdI. Desativar temporariamente para confirmar se e a causa. Corrigir a logica do BAdI.

### Cenario 5: Erro Suprimido pelo BAdI error_handling

**Sintoma**: Erro de calculo de impostos nao e reportado. Pricing aparenta estar OK mas valores estao incorretos ou zerados.

**Causa raiz**: Implementacao do BAdI `TXS_BADI_CONTROLLER::ERROR_HANDLING` retornando `r_overwrite_standard = abap_true`, suprimindo o tratamento standard de erros.

**Debug**:
1. Breakpoint em `CL_TXS_CONTROLLER::IF_TXS_CONTROLLER~EXECUTE` (linhas 507-519)
2. Verificar `lv_overwrite_standard` apos a chamada do BAdI
3. Se `abap_true`: o erro foi suprimido -- nenhum log SLG1 gravado, `prsok` nao marcado como false

**Solucao**: Revisar implementacao do BAdI `error_handling`. Garantir que erros criticos nao sejam suprimidos.

## Fluxo de Execucao -- Fases 7-8

```
PROCESS_CONDITION_RECORD (para cada condition)
  |
  +-- Se TXSC: call_txs_from_pricing_formula()
  |     +-- [super] initialize() -> build_payload() -> calculate_taxes()
  |     +-- [BR] record_lfd_taxes()         <<<< FASE 8.1
  |           +-- CL_TXS_LFD_KEY_DETERMINER (item + document number)
  |           +-- CL_TXS_TAX_VALUES_BR (valores fiscais)
  |           +-- CL_TXS_LFD_RECORDER::FILL_LFD_TAXES (grava GSLOG_LOCAL_FISCAL_DATA)
  |
  +-- Se outra condition: map_txs_to_pricing_formula()
  |
  +-- CHECK_MAPPING (condition)              <<<< FASE 7
        +-- Se TXSC: reset mapped, return
        +-- Se mapped = true: return (OK)
        +-- Se kinak = 'Z' e is_last_condition: ERRO mapping_condition_error
        +-- Se kinak != 'Z': mapped = true (OK)
  |
  [fim do PROCESS_CONDITION_RECORD]

IF_TXS_CONTROLLER~EXECUTE (nivel externo)
  |
  +-- TRY: process_condition_record(...)
  |
  +-- CATCH cx_txs_exception               <<<< FASE 8.3
  |     +-- BAdI error_handling (pode suprimir)
  |     +-- Se overwrite = false:
  |           +-- clear_resp_buffer_by_error() (se TXSC)
  |           +-- add_app_log() (SLG1)
  |           +-- handle_pricing_error() (prsok = false, kinak = 'X')
  |
  +-- BAdI post_processing                  <<<< FASE 8.2
  |     +-- Pode alterar c_pricing_item, c_pricing_condition_record, c_pricing_condition_value
  |
  +-- reset_pricing_item()                  <<<< FASE 8.6 (BR: limpa MATNR subcontratacao)
```

## Checklist do Consultor -- Pos-Processamento

### Verificacao de Mapeamento (Fase 7)
- [ ] Pelo menos uma condition mapeada com sucesso (`mapped = abap_true`)
- [ ] Todas as condition types TXS tem formula 279 atribuida na pricing procedure
- [ ] Tabela `J_1BNFTXCOND` configurada para os condition types usados
- [ ] Tax Engine retorna tax types correspondentes as conditions configuradas

### Pricing Status (Fase 8)
- [ ] `prsok = abap_true` (pricing OK)
- [ ] Nenhuma condition com `KINAK = 'X'` (via_formulas_or_incorrect)
- [ ] SLG1 sem erros no objeto `TXS_LOG`

### Dados Fiscais (Fase 8 -- Brasil)
- [ ] LFD preenchido (`GSLOG_LOCAL_FISCAL_DATA` com valores fiscais para o documento)
- [ ] `CL_TXS_CONTROLLER_BR` sendo instanciado (country specialization ativa)
- [ ] `CL_TXS_LFD_KEY_DETERMINER` retornando chaves corretas (item + document number)
- [ ] `CL_TXS_TAX_VALUES_BR::GET_TAX_VALUES` retornando valores calculados

### BAdIs e Extensibilidade (Fase 8)
- [ ] Nenhum BAdI `post_processing` alterando resultados inesperadamente
- [ ] Nenhum BAdI `error_handling` suprimindo erros criticos (`r_overwrite_standard = false`)
- [ ] Se houver BAdIs ativos: comportamento documentado e validado

### Application Log (SLG1)
- [ ] Verificar SLG1 com filtro: Objeto = `TXS_LOG`, Data = hoje, Usuario = usuario de teste
- [ ] Se houver erros: analisar detalhes (clicar na mensagem para ver payload + response JSON)
- [ ] Correlation ID (`X-CorrelationID`) pode ser usado para rastrear no Tax Engine
