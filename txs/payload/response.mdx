## Visao Geral

A **TXS Response** e a resposta retornada pelo motor externo de calculo de impostos (Tax Engine) apos processar o payload enviado pelo SAP S/4HANA Cloud. Ela contem os valores de impostos calculados, organizados hierarquicamente:

```
TXS_RESPONSE (raiz)
  +-- tax_lines[] (TXS_TAX_LINE â€” um por item da transacao)
  |     +-- tax_values[] (TXS_TAX_VALUE â€” um por tipo de imposto no item)
  |           +-- tax_attributes[] (TXS_TAX_ATTRIBUTE â€” atributos fiscais: CST, CENQ, etc.)
  +-- warning[] (TXS_TAX_WARNING â€” alertas do motor)
```

**Fluxo**: O payload e enviado via HTTP POST pela classe `CL_TXS_COMMUNICATION`. A resposta JSON e desserializada para a estrutura `TXS_RESPONSE` e armazenada no Response Buffer (`CL_TXS_RESPONSE_BUFFER`) para reutilizacao pelas demais condition types do pricing.

## 1. Estrutura: TXS_RESPONSE (Raiz da Resposta)

**Fonte**: `incoming/txs_response.abap`

Estrutura ABAP anotada com `@AbapCatalog.enhancement.category : #EXTENSIBLE_ANY`, permitindo extensoes via custom fields.

| Campo | Tipo ABAP | Descricao | Uso no Mapeamento |
|-------|-----------|-----------|-------------------|
| `id` | `abap.string(0)` | Transaction ID â€” correlaciona com `TXS_PAYLOAD-id` enviado no request | Identificacao unica da transacao; armazenado no buffer como `trace_guid` |
| `total` | `abap.string(0)` | Valor total a pagar/receber (incluindo impostos) | Mapeado para condition TAAS_TOTAL via `MAP_COND_TAAS_TOTAL` (KWERT = total, KBETR = 1000). Fonte: `cl_txs_conditions_mapper_br.abap:486-487` |
| `inclusive` | `abap.string(0)` | Flag: imposto ja incluido no valor bruto (`true`/`false`) | Indica se o preco informado ja inclui impostos |
| `sub_total` | `abap.string(0)` | Valor base antes dos impostos | Mapeado para condition TAAS_SUBTOTAL via `MAP_COND_TAAS_SUBTOTAL` (KWERT = sub_total, KBETR = total_tax). Fonte: `cl_txs_conditions_mapper_br.abap:484-485` |
| `total_tax` | `abap.string(0)` | Soma total de impostos de todos os itens | Usado no mapeamento do TAAS_SUBTOTAL como KBETR |
| `total_withholding_tax` | `abap.string(0)` | Total de impostos retidos na fonte (WHT) | Soma de IR, CSLL, PIS-WHT, COFINS-WHT, INSS-WHT |
| `country` | `abap.string(0)` | Codigo do pais tributavel | Para Brasil: `BR` |
| `date` | `abap.string(0)` | Timestamp da resposta (data/hora de calculo) | Armazenado no buffer; na geracao do hash, truncado para 10 caracteres (apenas data) |
| `tax_lines` | `txs_tax_line_tty` | Tabela de linhas de imposto (uma por item da transacao) | Estrutura `TXS_TAX_LINE` â€” ver secao 2 |
| `warning` | `txs_tax_warning_tty` | Tabela de alertas retornados pelo motor | Estrutura `TXS_TAX_WARNING` â€” ver secao 6 |

**Observacoes**:
- Todos os campos sao do tipo `abap.string(0)` (string dinamica), pois o JSON retornado pelo tax engine e flexivel. A conversao para tipos numericos ocorre nas classes de mapeamento.
- O campo `id` retornado deve ser identico ao `id` enviado no payload, permitindo correlacao request-response.

## 2. Estrutura: TXS_TAX_LINE (Por Item)

**Fonte**: `incoming/txs_tax_line.abap`

Cada `tax_line` corresponde a um item do documento SAP (pedido de compra, ordem de venda, etc.). A correlacao com o item SAP e feita pelo campo `id`.

| Campo | Tipo ABAP | Descricao | Uso no Mapeamento |
|-------|-----------|-----------|-------------------|
| `id` | `abap.string(0)` | Identificador unico do item â€” correlaciona com `TXS_ITEM-id` enviado no request | Usado como chave para localizar o tax_value correto no `CL_TXS_TAX_TYPES_VALUES_BR::BUILD` |
| `total_tax` | `abap.string(0)` | Soma de todos os impostos calculados para este item | Valor consolidado; detalhes por imposto em `tax_values` |
| `taxcode` | `abap.string(0)` | Codigo de imposto determinado pelo tax engine (codigo externo) | Mapeado para o tax code SAP (S4) via tabela `TXSMAPEXTTOERP`. Armazenado no buffer como `ext_tax_code` |
| `total_rate` | `abap.string(0)` | Aliquota total combinada de todos os impostos do item | Soma das aliquotas individuais |
| `total_withholding_tax` | `abap.string(0)` | Total de impostos retidos na fonte para este item | Soma de WHT taxes (PIS-WHT, COFINS-WHT, CSLL-WHT, IR-WHT, INSS-WHT) |
| `total_withholding_tax_rate` | `abap.string(0)` | Aliquota total de retencao na fonte | Soma das aliquotas de WHT |
| `withholding_tax_code` | `abap.string(0)` | Codigo de imposto retido na fonte | Codigo WHT determinado pelo engine |
| `tax_values` | `txs_tax_value_tty` | Tabela de valores por tipo de imposto | Estrutura `TXS_TAX_VALUE` â€” ver secao 3 |

**Importante**: O campo `taxcode` e o codigo externo do tax engine, **nao** o tax code SAP (MWSKZ). O mapeamento externo-para-SAP ocorre via tabela de configuracao `TXSMAPEXTTOERP`.

## 3. Estrutura: TXS_TAX_VALUE â€” A Mais Critica

**Fonte**: `incoming/txs_tax_value.abap`

Esta e a estrutura central da response. Cada `tax_value` representa um imposto individual calculado (ICMS, IPI, PIS, etc.) para um item especifico. E aqui que estao os valores que serao mapeados para as condition records do pricing SAP (KOMV).

| Campo | Tipo ABAP | Descricao | Mapeamento para KOMV | Detalhe |
|-------|-----------|-----------|---------------------|---------|
| `level` | `abap.string(0)` | Nivel na hierarquia de impostos (para cenarios de imposto-sobre-imposto) | Uso interno no parse | Ex: IPI nivel 1, ICMS-ST nivel 2 (IPI entra na base do ICMS-ST) |
| `tax_type_code` | `abap.string(0)` | Codigo que identifica o imposto (ICMS, IPI, PIS, etc.) | Chave de dispatch â€” determina qual metodo `BUILD_MM_*` / `BUILD_SD_*` sera chamado | Ver tabela completa na secao 4 |
| `name` | `abap.string(0)` | Nome legivel do tipo de imposto | Nao mapeado | Apenas descritivo |
| `rate` | `abap.string(0)` | Aliquota do imposto (%) | `KOMV-KBETR` = rate x 10 | Fator de conversao: `con_factor = 10`. Fonte: `if_logbr_ft_txs_constants.abap:73` |
| `taxable` | `abap.string(0)` | Base de calculo (valor sobre o qual o imposto incide) | `KOMV-KAWRT` | Valor base para o calculo do imposto |
| `exempted_base_percent` | `abap.string(0)` | Percentual da base isenta (reducao de base ICMS) | `KOMV-KBETR` = (100 - exempt%) x 10 | Usado em `MAP_COND_BASE_REDUCTION_RATE`. Fonte: `cl_txs_conditions_mapper_br.abap:466-467` |
| `exempted_base_amount` | `abap.string(0)` | Valor da base isenta em moeda | `KOMV-KWERT` | Via `MAP_COND_EXCLUDED_BASE`. Fonte: `cl_txs_conditions_mapper_br.abap:468-469` |
| `other_base_amount` | `abap.string(0)` | Outra base (pauta fiscal, valor de mercado) | `KOMV-KWERT` | Via `MAP_COND_OTHER_BASE`. Fonte: `cl_txs_conditions_mapper_br.abap:470-471` |
| `value` | `abap.string(0)` | Valor calculado do imposto (incluindo parte nao dedutivel) | `KOMV-KWERT` | Via `MAP_COND_AMOUNT`. Fonte: `cl_txs_conditions_mapper_br.abap:462-463` |
| `non_deductible_tax_rate` | `abap.string(0)` | Aliquota do imposto nao dedutivel (nao recuperavel) | Sem mapeamento direto | Usado na logica de credito fiscal |
| `non_deductible_tax_amount` | `abap.string(0)` | Valor do imposto nao dedutivel | Sem mapeamento direto | Diferenca: value - deductible_tax_amount |
| `deductible_tax_amount` | `abap.string(0)` | Valor do imposto dedutivel (credito fiscal) | Usado na logica `HAS_CREDIT` | `HAS_CREDIT` retorna `true` se `deductible_tax_amount` > 0 |
| `tax_attributes` | `txs_tax_attribute_tty` | Tabela de atributos fiscais (CST, CENQ, etc.) | Ver secao 5 | Contem informacoes adicionais por tipo de imposto |
| `jurisdiction` | `abap.string(0)` | Nome da jurisdicao fiscal | Nao mapeado diretamente | Ex: "Sao Paulo" |
| `jurisdiction_code` | `abap.string(0)` | Codigo da jurisdicao fiscal (UF) | Mapeado via `CL_TXS_JURISDICTION_MAPPER` | Para BR: codigo do estado (SP, RJ, MG...) |
| `due_category` | `abap.string(0)` | Categoria de recolhimento: `P` = payable (a pagar) | Constante: `DUE_CATEGORY_PAYABLE = 'P'`. Fonte: `if_txs_constants.abap:151` | Indica se o imposto e a recolher ou a recuperar |
| `is_tax_deferred` | `abap.string(0)` | Flag: imposto diferido (`true`/`false`) | Tratado na logica de ICMS-DP | Diferimento parcial de ICMS |
| `withholding_relevant` | `abap.string(0)` | Flag: imposto relevante para retencao na fonte | Determina se o tax_value participa do WHT | `true` para PIS-WHT, COFINS-WHT, CSLL-WHT, etc. |
| `full_rate` | `abap.string(0)` | Aliquota original antes de reducoes | Sem mapeamento direto | Util para analise: rate = full_rate quando nao ha reducao |

### Fatores de Conversao Response -> KOMV

Verificados no codigo-fonte:

## 7. Processamento da Response

### 7.1 Fluxo Completo

O processamento da response segue estas etapas:

```
1. HTTP POST (payload JSON) ---> Tax Engine
2. Tax Engine processa e retorna JSON
3. CL_TXS_COMMUNICATION::FETCH_RESPONSE
   |-- Verifica HTTP status (200 = OK)
   |-- Desserializa JSON -> TXS_RESPONSE
   |-- Cria instancia CL_TXS_RESPONSE
4. CL_TXS_RESPONSE_BUFFER::ADD_RESPONSE
   |-- Calcula SHA1 hash do payload (sem id/time)
   |-- Armazena response no buffer (singleton)
5. CL_TXS_TAX_TYPES_VALUES_BR::BUILD
   |-- Para cada tax_line -> para cada tax_value:
   |     Dispatch por tax_type_code -> BUILD_MM_* ou BUILD_SD_*
   |-- Preenche tabela interna de tax types mapeados
6. CL_TXS_CONDITIONS_MAPPER_BR::MAP_TAX_TYPE_TO_CONDITION
   |-- Para cada condition type na J_1BNFTXCOND:
   |     Mapeia tax_value -> KOMV (pricing condition record)
```

### 7.2 Desserializacao

**Classe**: `CL_TXS_COMMUNICATION`
**Metodo**: `FETCH_RESPONSE` (fonte: `cl_txs_communication.abap:211`)

```abap
method FETCH_RESPONSE.
  data(lv_http_status) = client->get_status( ).
  data(lr_response_entity) = client->get_response_entity( ).

  if lv_http_status = con_http_status_ok.
    " 1. camelCase -> snake_case
    data(json_string) = json_tool->rename_properties_to_field(
      lr_response_entity->get_string_data( ) ).

    " 2. Desserializa e cria instancia
    response = cl_txs_response=>get_instance(
      i_country         = country
      i_response_json   = json_string
      i_payload         = payload->get_payload( )
      i_pricing_context = pricing_context ).
  else.
    " Erro HTTP -> exception CX_TXS_EXCEPTION
    raise exception type cx_txs_exception ...
  endif.
endmethod.
```

**Conversao JSON**: O metodo `CL_TXS_JSON_SERIALIZER::RENAME_PROPERTIES_TO_FIELD` converte nomes de propriedades JSON de camelCase para snake_case (formato dos campos ABAP). Fonte: `cl_txs_json_serializer.abap:954`.

**Desserializacao**: `CL_TXS_RESPONSE::DESERIALIZE_RESPONSE` usa `pretty_name = json_tool->pretty_mode-camel_case` para o parser. Fonte: `cl_txs_response.abap:124-137`.

### 7.3 Parse por Tax Type

**Classe**: `CL_TXS_TAX_TYPES_VALUES_BR`
**Metodo principal**: `BUILD_MM_PER_TAX_VALUE` (MM) / `BUILD_SD_PER_TAX_VALUE` (SD)

O dispatch e feito via `cl_txs_internal_codes=>get_instance( )->is_value_in_internal_code()`, comparando o `tax_type_code` retornado com os internal codes configurados. Isso permite que o mapeamento tax_type_code -> imposto seja configuravel, sem hardcoding de strings.

Cada metodo `BUILD_MM_*` / `BUILD_SD_*` recebe:
- `I_ID` (string): ID do item
- `I_TAX` (TXS_TAX_VALUE): Valores do imposto
- `I_ATTRIBUTES` (ref to IF_TXS_TAX_ATTRIBUTES): Acesso aos tax_attributes

E chama `APPEND_TAX_TYPE` para adicionar a tax type mapeada na tabela interna.

### 7.4 Response Buffer

**Classe**: `CL_TXS_RESPONSE_BUFFER` (singleton)
**Interface**: `IF_TXS_RESPONSE_BUFFER`
**Fonte**: `incoming/cl_txs_response_buffer.abap`, `incoming/if_txs_response_buffer.abap`

O buffer armazena responses para reutilizacao. A chamada HTTP ao tax engine ocorre apenas uma vez (na condition type **TXSC**). Todas as demais condition types do pricing procedure leem do buffer.

#### Estrutura do Buffer

**Fonte**: `if_txs_response_buffer.abap:6-23`

| Campo | Tipo | Descricao |
|-------|------|-----------|
| `request_hash` | `string` | Hash SHA1 do payload serializado (chave primaria) |
| `document_item_key` | `char16` | Chave do item do documento (chave primaria) |
| `timestamp_service_call` | `timestampl` | Timestamp da chamada ao servico |
| `timestamp_buffer_read` | `timestampl` | Timestamp da ultima leitura do buffer |
| `date` | `string` | Data do request original |
| `response` | `string` | JSON da response (string completa) |
| `request` | `string` | JSON do request serializado |
| `ext_tax_code` | `txs_ext_tax_code` | Tax code externo (do engine) |
| `s4_tax_code` | `mwskz` | Tax code SAP (mapeado) |
| `tax_date` | `prsdt` | Data fiscal |
| `trace_guid` | `txs_guid` | GUID de rastreamento |
| `document_item_id` | `txs_doc_item` | ID do item no documento |
| `reference` | `char16` | Referencia do item |
| `jurisdiction_code` | `txjcd` | Codigo de jurisdicao |
| `kappl` | `kappl` | Aplicacao (TX = MM, V = SD) |
| `pricing_context` | `ref to if_txs_pricing_context` | Referencia ao contexto de pricing |

#### Calculo do Hash

**Fonte**: `cl_txs_response_buffer.abap:70-97`

O hash e calculado com SHA1 sobre o payload serializado, **removendo** o campo `id` e truncando `date` para 10 caracteres (somente data, sem hora). Isso garante que payloads identicos (mesmo item, mesmos dados fiscais) produzam o mesmo hash, mesmo em chamadas diferentes.

```abap
" Prepara payload para hash (remove volatilidade)
clear ls_request-id.
ls_request-date = ls_request-date(10).  " Apenas data

" Serializa e calcula SHA1
data(lv_payload_serialized) = serialize_request( ls_request ).
r_hash = calculate_hash( lv_payload_serialized ).
```

#### Lookup por Invoice vs. Outros

**Fonte**: `cl_txs_response_buffer.abap:285-291`

Para documentos de invoice, a chave de busca no buffer inclui `document_item_id` (TAXPS). Para outros documentos, usa `document_item_key`:

```abap
IF cl_txs_business_transaction=>get_instance( )->is_invoice( ).
  READ TABLE response_buffer WITH KEY
    request_hash = lv_request_hash
    document_item_id = i_pricing_context->get_item( )-taxps.
ELSE.
  READ TABLE response_buffer WITH KEY
    request_hash = lv_request_hash
    document_item_key = lv_doc_ref.
ENDIF.
```

#### Feature Toggle

**Fonte**: `cl_txs_controller.abap:203`

O feature toggle `GLO_LOG_BR_EXT_JSON_RESP` (tipo `ftg_id`) controla o metodo de armazenamento da response no buffer. Quando ativo, utiliza o metodo `add_response()` do controller; quando inativo, utiliza o metodo padrao `IF_TXS_RESPONSE_BUFFER~ADD_RESPONSE`:

```abap
CONSTANTS ftg_br2_toggle_resconsp TYPE ftg_id VALUE 'GLO_LOG_BR_EXT_JSON_RESP'.
IF cl_ftg_runtime_state=>is_on( ftg_br2_toggle_resconsp ).
  add_response( ).
ELSE.
  lr_response_buffer->if_txs_response_buffer~add_response( ... ).
ENDIF.
```

#### BAdI do Buffer

O buffer suporta o BAdI `TXS_BADI_RESPONSE_BUFFER` com dois pontos de extensao:
- `prepare_request_to_be_hashed`: Permite alterar o request antes do calculo do hash (fonte: `cl_txs_response_buffer.abap:121`)
- `get_response`: Permite interceptar a leitura do buffer (fonte: `cl_txs_response_buffer.abap:374`)

### Dica de Debug â€” Sequencia Recomendada

1. **Verificar a chamada HTTP**: Breakpoint em `FETCH_RESPONSE` para ver o status HTTP e o JSON bruto
2. **Verificar desserializacao**: Breakpoint em `DESERIALIZE_RESPONSE` para ver se a estrutura foi populada corretamente
3. **Verificar dispatch**: Breakpoint em `BUILD_MM_PER_TAX_VALUE` ou `BUILD_SD_PER_TAX_VALUE` para ver qual metodo sera chamado para cada tax_type_code
4. **Verificar valores especificos**: Breakpoint no metodo `BUILD_MM_*` correspondente ao imposto com problema
5. **Verificar mapeamento KOMV**: Breakpoint em `MAP_TAX_TYPE_TO_CONDITION` para ver os valores finais no pricing
