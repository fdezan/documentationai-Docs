## Como Usar Este Guia

1. Abra o Eclipse ADT e navegue ate a classe indicada
2. Coloque breakpoint no metodo indicado
3. Execute o cenario SAP (Pedido de Compra, Billing, etc.)
4. Inspecione as variaveis indicadas na coluna "O que verificar"

**Pre-requisito**: O TXS usa a formula 279 na pricing procedure para acionar o calculo. Todas as condition types TXS (TXSC + conditions de imposto) devem ter esta formula atribuida.
Fonte: `cl_txs_condition_processing.abap:71` — `constants con_formula_279 type kofrm value 0000279`

**Condition de service call**: O TXS usa a condition type **TXSC** para disparar a chamada HTTP ao Tax Engine.
Fonte: `if_txs_constants.abap:146` — `constants SERVICE_CALL type KSCHA value 'TXSC'`

**Inativacao de condition**: Quando o TXS esta inativo, a condition TXSC recebe KINAK = 'Z'.
Fonte: `if_txs_constants.abap:83` — `constants TXSC_CONDITION_INACTIVE type KINAK value 'Z'`

## Fase 1 — Entrada e Validacoes

### Debug Point 1: Preenchimento LFD (Local Fiscal Data)

- **Classe**: `CL_LOGBR_FILL_LFD_ON_PRINCING`
- **Metodo**: `RUN`
- **Onde no codigo**: Chamado no inicio de `IF_TXS_CONTROLLER~EXECUTE`
  Fonte: `cl_txs_controller.abap:404-408`:
  ```abap
  DATA(fill_pricing_with_lfd) = NEW cl_logbr_fill_lfd_on_princing( ).
  c_pricing_item = fill_pricing_with_lfd->run(
    i_pricing_header = i_pricing_header
    i_pricing_item   = c_pricing_item
  ).
  ```
- **O que verificar**:
  - `c_pricing_item` apos o retorno — campos fiscais preenchidos:
    - `STEUC` (NCM — classificacao fiscal do material)
    - `TXJCD` (Tax Jurisdiction Code — codigo IBGE do municipio)
    - `J_1BMATORG` (Material Origin)
    - `J_1BMATUSE` (Material Usage)
- **Problema diagnosticado**: NCM vazio, CFOP errado, dados fiscais ausentes no item de pricing
- **Acao se falhar**: Verificar cadastro de material (`MARA-STEUC`, `MARC-STEUC`) e dados de endereco da planta/empresa

---

### Debug Point 2: Verificacao de Ativacao TXS

- **Classe**: `CL_TXS_SWITCH`
- **Metodo**: `IS_ACTIVE`
- **Onde no codigo**: Chamado em `IF_TXS_CONTROLLER~EXECUTE` para decidir se o TXS processa
  Fonte: `cl_txs_controller.abap:469-477`:
  ```abap
  IF cl_txs_switch=>get_instance( )->is_active(
       i_application = i_pricing_header-kappl
       i_pricing_procedure = i_pricing_header-kalsm
       i_country = i_countries-land1
       i_company_code = i_pricing_header-bukrs ) = abap_false.
    IF lr_controller->is_service_call_cond_type( c_pricing_condition_record-kschl ).
      c_pricing_condition_record-kinak = if_txs_constants=>txsc_condition_inactive.
    ENDIF.
    RETURN.
  ENDIF.
  ```
- **O que verificar**:
  - `r_active` — resultado final (true/false)
  - `active` — conexao ativa (verificada no `CONSTRUCTOR` via `cl_txs_conn_config_factory`)
    Fonte: `cl_txs_switch.abap:76-80`:
    ```abap
    data(lr_connection_config) = cl_txs_conn_config_factory=>get_instance( )->get_connection_config( ).
    if lr_connection_config is bound.
      active = lr_connection_config->is_connection_active( ).
    endif.
    ```
  - `is_integration_active()` — verifica tabela `TXS_ACTIVATION` (application + pricing procedure)
    Fonte: `cl_txs_switch.abap:113-115`:
    ```abap
    r_active = cl_txs_activation_dao=>get_instance( )->get_tax_service_active_status(
      i_application  = i_application
      i_pricing_name = i_pricing_procedure ).
    ```
  - `check_activation_exceptions()` — verifica tabela `TXS_ACTIVATION_EXCEPTION` (empresa/planta)
    Fonte: `cl_txs_switch.abap:54-70`:
    ```abap
    data(ls_activation_exception) = lr_activation_dao->get_tax_service_except_entry( ... ).
    r_active = xsdbool( ls_activation_exception is initial or
       ls_activation_exception-calculation_engine = if_txs_constants=>calculation_engine-partner_engine ).
    ```
  - Activation Control V2: `cl_txs_activation_control=>get_instance( )->if_txs_activation_control~is_integration_active()`
    Fonte: `cl_txs_switch.abap:124-128`
- **Problema diagnosticado**: TXS nao e chamado, condition TXSC fica com KINAK='Z'
- **Acao se falhar**:
  - Verificar tabela `TXS_ACTIVATION` (chave: application `V`/`TX` + pricing procedure)
  - Verificar tabela `TXS_ACTIVATION_EXCEPTION` (empresa/planta especifica)
  - Verificar se `calculation_engine` = `'P'` (Partner Engine)
    Fonte: `if_txs_constants.abap:200` — `partner_engine type txs_calculation_engine value 'P'`

### Debug Point 3: Pricing Context

- **Classe**: `CL_TXS_PRICING_CONTEXT`
- **Metodo**: `CONSTRUCTOR`
- **Onde no codigo**: Criado em `IF_TXS_CONTROLLER~EXECUTE` via `CREATE_PRICING_CONTEXT`
  Fonte: `cl_txs_controller.abap:455-463`:
  ```abap
  lr_controller->set_pricing_context( create_pricing_context(
    i_pricing_header            = i_pricing_header
    i_pricing_item              = c_pricing_item
    i_pricing_condition_records = i_pricing_condition_records
    i_plants_or_branches        = i_plants_or_branches
    i_countries                 = i_countries
    i_transaction_code          = i_transaction_code
    i_local_time                = i_local_time
    i_processing_moment         = i_pricing_processing_moment
    i_group_condition           = i_group_condition ) ).
  ```
- **O que verificar**:
  - `i_pricing_header` (KOMK) — cabecalho do pricing: `KAPPL` (application), `KALSM` (pricing procedure), `BUKRS` (empresa), `BELNR` (documento), `TRTYP` (tipo transacao)
  - `i_pricing_item` (KOMP) — item do pricing: `KPOSN` (posicao), `MATNR` (material), `MWSKZ` (tax code), `STEUC` (NCM), `EVRTN`/`EVRTP` (referencia PO)
  - `i_pricing_condition_records` (TXS_KOMV_INDEX_TTY) — todas as conditions do pricing
  - `i_plants_or_branches` (T001W) — planta/filial
  - `i_countries` (T005) — pais (INTCA, LAND1)
  - `i_transaction_code` — tcode corrente (ME21N, VA01, MIRO, etc.)
- **Problema diagnosticado**: Dados de entrada incorretos para todo o pipeline (material vazio, planta errada, application incorreta)

## Fase 2 — Decisao: Service Call vs Mapping

### Debug Point 4: Decisao Service Call vs Buffer Read

- **Classe**: `CL_TXS_CONTROLLER`
- **Metodo**: `PROCESS_CONDITION_RECORD`
- **Onde no codigo**:
  Fonte: `cl_txs_controller.abap:647-669`:
  ```abap
  method process_condition_record.
    data(lr_payload_type) = cl_txs_payload_type=>get_instance( pricing_context->get_country( ) ).
    data(lv_payload_type) = lr_payload_type->determine_payload_type( pricing_context ).
    ...
    if is_service_call_cond_type( c_pricing_condition_record-kschl ).
      call_txs_from_pricing_formula( ).
    else.
      map_txs_to_pricing_formula( ... ).
    endif.
    cl_txs_condition_processing=>get_instance( country )->check_mapping( ... ).
  endmethod.
  ```
- **O que verificar**:
  - `c_pricing_condition_record-kschl` — a condition type atual
  - `is_service_call_cond_type()` — retorna `true` se for TXSC (service call), `false` para outras conditions (que fazem buffer read)
    Fonte: `cl_txs_controller.abap:612-617` — verifica via `cl_txs_internal_codes` se o valor esta configurado como `condition_type-tax_service_integration`
  - Se TXSC: chama `call_txs_from_pricing_formula()` (HTTP call)
  - Se outra condition: chama `map_txs_to_pricing_formula()` (leitura do buffer)
  - Verificacao de material vazio: se `MATNR` esta vazio e nao e Canada e payload type e V1, o metodo faz `RETURN` sem processar
    Fonte: `cl_txs_controller.abap:652-656`
- **Problema diagnosticado**: Condition errada triggerando ou nao a chamada HTTP, item sem material nao sendo processado

### Debug Point 5: Tipo de Payload

- **Classe**: `CL_TXS_PAYLOAD_TYPE`
- **Metodo**: `DETERMINE_PAYLOAD_TYPE`
- **Onde no codigo**: Chamado em `PROCESS_CONDITION_RECORD` para determinar o tipo de payload
  Fonte: `cl_txs_controller.abap:649-650`:
  ```abap
  data(lr_payload_type) = cl_txs_payload_type=>get_instance( pricing_context->get_country( ) ).
  data(lv_payload_type) = lr_payload_type->determine_payload_type( pricing_context ).
  ```
- **O que verificar**:
  - Resultado retornado — determina a classe de payload usada:
    - **SD + integracao V2 ativa**: `CL_TXS_REQUEST_V2` (V2 com parties no item, extended fields)
    - **MM + auto tax code**: `CL_TXS_PAYLOAD` (V1 standard)
    - **MM default**: `CL_TXS_DIRECT_CALC_PAYLOAD` (calculo direto)
  - `pricing_context->get_payload_type()` apos determinacao — valor `'V2'` ou outro
    Fonte: `if_txs_constants.abap:204` — `PAYLOAD_VERSION_V2 type string value 'V2'`
- **Problema diagnosticado**: Payload V1 quando deveria ser V2 (ou vice-versa), faltam campos V2 no request

## Fase 3 — Build do Payload

### Debug Point 6: Header do Payload

- **Classe**: `CL_TXS_PAYLOAD_BR`
- **Metodo**: `BUILD_HEADER`
- **Onde no codigo**: Chamado durante `payload->build()` em `CL_TXS_CONTROLLER.BUILD_PAYLOAD`
  Fonte: `cl_txs_controller.abap:169-173`:
  ```abap
  method build_payload.
    payload = cl_txs_payload_factory=>get_instance( )->get_payload( pricing_context ).
    payload->set_pricing_context( pricing_context ).
    payload->build( ).
  endmethod.
  ```
- **O que verificar**:
  - `payload-id` — UUID unico da requisicao (usado como X-CorrelationID no HTTP)
  - `payload-sale_or_purchase` — `'S'` (SD) ou `'P'` (MM)
  - `payload-operation_nature_code` — tipo de operacao: `'P'` (Purchase), `'S'` (Sale), `'PI'` (Purchase Invoice), `'SI'` (Sales Invoice), `'STO'` (Stock Transfer), `'CI'`/`'CS'` (Convergent Invoice)
  - `payload-gross_or_net` — determina se preco base e bruto ou liquido
  - `payload-date` — data do pricing
  - `payload-currency` — moeda
  - `payload-company_id` — CNPJ da empresa
- **Problema diagnosticado**: Operation nature errado, data incorreta, gross/net invertido, empresa errada

---

### Debug Point 7: Preco Unitario

- **Classe**: `CL_TXS_ITEMS_BR`
- **Metodo**: `GET_UNIT_PRICE` (despacha para `GET_UNIT_PRICE_MM` ou `GET_UNIT_PRICE_SD`)
- **O que verificar**:
  - `lv_unit_price` — preco unitario do item (deve ser > 0)
  - `ls_pricing_value-kwert` — valor base da condition usada como referencia de preco
  - Cenario MM vs SD: metodo correto sendo chamado
- **Problema diagnosticado**: Preco base zero ou incorreto no payload enviado ao Tax Engine

---

### Debug Point 8: Quantidade

- **Classe**: `CL_TXS_ITEMS_BR`
- **Metodo**: `GET_QUANTITY` / `GET_QUANTITY_TAXABLE`
- **O que verificar**:
  - `result` — quantidade do item
  - Feature toggle `GLO_LOG_BR_MAA_NF_AFD_MAPPING` — afeta quantidade em cenarios de invoice com MAA
- **Problema diagnosticado**: Quantidade incorreta (especialmente em invoice com AFD/MAA)

---

### Debug Point 9: NCM e Classificacoes Fiscais

- **Classe**: `CL_TXS_ITEM_CLASSIFICATIONS_BR`
- **Metodo**: `GET_MATERIAL_NCM` / `APPEND_ITEM_CLASSIFICATIONS`
- **O que verificar**:
  - `isc_code` — NCM do material (8 digitos, sem pontos — ex: `84719012`)
  - `pricing_context->get_item()->steuc` — valor bruto do NCM vindo do KOMP
  - CEST, EAN — outras classificacoes fiscais adicionadas
  - `item_classifications[]` — lista completa de classificacoes do item
- **Segundo breakpoint**: `CL_TXS_MATERIAL_INFO_FORMATTER`, metodo `FORMAT_NCM`
  - Remove pontos, toma 8 caracteres
- **Problema diagnosticado**: NCM vazio ou formato errado (pontos nao removidos, menos de 8 digitos)
- **Acao se falhar**: Verificar cadastro de material `MARC-STEUC`; para STO verificar planta de origem

---

### Debug Point 10: CNPJ

- **Classe**: `CL_TXS_PARTY_BR`
- **Metodo**: `GET_VALUE_CNPJ`
- **O que verificar**:
  - `tax_number` — CNPJ numerico (14 digitos, sem formatacao)
  - Origem: empresa (T001/J_1BBRANCH) ou parceiro de negocios (LFA1/KNA1)
- **Problema diagnosticado**: CNPJ ausente ou com formatacao (pontos/barras)

---

### Debug Point 11: CNAE

- **Classe**: `CL_TXS_CNAE_BR`
- **Metodo**: Depende do cenario:
  - MM compra material: `CNAE_FOR_MM_BP_MAT` + `CNAE_FOR_MM_PLANT_MAT`
  - MM compra servico: `CNAE_FOR_MM_BP_SERV` + `CNAE_FOR_MM_PLANT_SERV`
  - SD venda material: `CNAE_FOR_SD_BP_MAT` + `CNAE_FOR_SD_PLANT_MAT`
- **O que verificar**:
  - Resultado CNAE retornado
  - Fonte do CNAE: dados mestre — `LFA1-CNAE` (fornecedor), `KNA1-CNAE` (cliente), `J_1BBRANCH-CNAE` (planta). ~~BRF+ descontinuado no TXS~~
- **Problema diagnosticado**: CNAE vazio (causa ISS incorreto, especialmente em servicos)
- **Acao se falhar**: Preencher CNAE nos dados mestre: `LFA1-CNAE`, `KNA1-CNAE` ou `J_1BBRANCH-CNAE`

---

### Debug Point 12: Locations (Enderecos)

- **Classe**: `CL_TXS_LOCATIONS_BR`
- **Metodo**: `ADD_LOCATIONS` (despacha internamente para cenario especifico)
- **O que verificar**:
  - `_locations[]` — lista de locations do payload:
    - `type` — `SHIP_FROM`, `SHIP_TO`, `CONTRACT_FROM`, `CONTRACT_TO`, `SERVICE_LOCATION`
      Fonte: `if_txs_constants.abap:131-138` (deprecados, agora via Internal Codes)
    - `state` — UF do estado (2 caracteres)
    - `country` — pais
    - `is_bp_tax_registered` — se o parceiro e contribuinte
  - Cenarios criticos: STO (ambas as plantas), intercompany, importacao
- **Problema diagnosticado**: UF incorreto, falta ship_from/to, B2B vs B2C errado, endereco de entrega diferente do endereco fiscal

## Fase 4 — Comunicacao HTTP

### Debug Point 13: HTTP POST

- **Classe**: `CL_TXS_COMMUNICATION`
- **Metodo**: `POST`
- **Onde no codigo**:
  Fonte: `cl_txs_communication.abap:330-348`:
  ```abap
  method post.
    try.
        client->if_rest_resource~post( request ).
      catch cx_rest_client_exception into data(lx_exception).
        data(lr_log_dao) = cl_txs_log_dao=>get_instance( ).
        lr_log_dao->add_std_exception_to_log( i_exception = lx_exception ).
        raise exception type cx_txs_exception
          exporting
            textid      = if_txs_const_exceptions=>http_client_error
            ...
    endtry.
  endmethod.
  ```
- **O que verificar**:
  - `lx_exception` — excecao de rede (`cx_rest_client_exception`)
  - Headers enviados:
    - `X-CorrelationID` = `payload->get_payload( )-id` (UUID)
      Fonte: `cl_txs_communication.abap:97-98`
    - `X-Routing-Key` = `partner_routing_key` (da TXS_ACTIVATION_EXCEPTION)
      Fonte: `cl_txs_communication.abap:100-101`
    - `Content-Type` = `application/json`
      Fonte: `cl_txs_communication.abap:92`
    - `Connection` = `keep-alive`
      Fonte: `cl_txs_communication.abap:94-95`
- **Segundo breakpoint**: Metodo `FETCH_RESPONSE` (ou `DESERIALIZE` em versoes anteriores)
  Fonte: `cl_txs_communication.abap:211-242`:
  - `lv_http_status` — se diferente de `200` (constante `con_http_status_ok`), gera excecao `http_server_error`
    Fonte: `cl_txs_communication.abap:49` — `data CON_HTTP_STATUS_OK type I value 200`
  - `error_response_json` — JSON de erro retornado pelo Tax Engine
- **Problema diagnosticado**: Timeout, erro de rede, Tax Engine offline, status HTTP != 200

- **Classe**: `CL_TXS_CONNECTION`
- **Metodo**: `GET_REST_CLIENT`

  ```abap
  endmethod.
  ```
- **Problema diagnosticado**: Logical Port nao configurado, token OAuth2 expirado, credenciais Client Credentials incorretas

## Fase 5 — Processamento da Response

### Debug Point 15: Parse por Tax Type

- **Classe**: `CL_TXS_TAX_TYPES_VALUES_BR`
- **Metodo**: Depende do cenario e imposto:
  - MM: `BUILD_MM_ICMS`, `BUILD_MM_IPI`, `BUILD_MM_PIS`, `BUILD_MM_COFINS`, `BUILD_MM_ISS`, etc.
  - SD: `BUILD_SD_ICMS`, `BUILD_SD_IPI`, `BUILD_SD_PIS`, `BUILD_SD_COFINS`, etc.
- **O que verificar**:
  - `i_tax` (TXS_TAX_VALUE) — estrutura retornada pelo Tax Engine:
    - `rate` — aliquota do imposto
    - `taxable` — base de calculo
    - `value` — valor calculado
    - `tax_type_code` — tipo de imposto (ICMS, IPI, PIS, COFINS, ISS, etc.)
    - `exempted_base_percent` — percentual de reducao de base
    - `exempted_base_amount` — valor de base isenta
- **Problema diagnosticado**: Aliquota ou valor incorreto retornado pelo Tax Engine, base de calculo zerada

---

### Debug Point 16: Response Buffer (Leitura)

- **Classe**: `CL_TXS_RESPONSE_BUFFER`
- **Metodo**: `IF_TXS_RESPONSE_BUFFER~GET_RESPONSE`
- **Onde no codigo**: Chamado em `CL_TXS_CONTROLLER.CALCULATE_TAXES` para tentar ler response do buffer antes de fazer HTTP call
  Fonte: `cl_txs_controller.abap:198-211`:
  ```abap
  try.
      response = lr_response_buffer->if_txs_response_buffer~get_response(
        i_request = payload i_pricing_context = pricing_context ).
    catch cx_txs_response_not_on_buffer.
      lr_communication->calculate_taxes( ).
      response = lr_communication->get_response( ).
      ...
  endtry.
  ```
- **O que verificar**:
  - `lv_request_hash` — hash SHA1 do payload usado como chave de busca no buffer
    Fonte: `cl_txs_response_buffer.abap:101-133` — metodo `CALCULATE_REQUEST_HASH`:
    - Limpa `id` e trunca `date` para 10 caracteres antes de calcular hash
    - BAdI `txs_badi_response_buffer->prepare_request_to_be_hashed` pode modificar o request antes do hash
  - Busca no buffer: por `request_hash` + `document_item_key` (ou `document_item_id` para invoice)
    Fonte: `cl_txs_response_buffer.abap:396-405`
  - Se `<ls_response_buffer>` nao atribuido: excecao `CX_TXS_RESPONSE_NOT_ON_BUFFER`
    Fonte: `cl_txs_response_buffer.abap:407-408`
  - Se `<ls_response_buffer>-response` vazio: excecao `CX_TXS_RESPONSE_NOT_ON_BUFFER`
    Fonte: `cl_txs_response_buffer.abap:417-419`
- **Problema diagnosticado**: Response nao encontrada no buffer (hash mismatch), a chamada HTTP da condition TXSC falhou na 1a condition

---

### Debug Point 16b: Response Buffer (Gravacao)

- **Classe**: `CL_TXS_RESPONSE_BUFFER`
- **Metodo**: `IF_TXS_RESPONSE_BUFFER~ADD_RESPONSE`
- **Onde no codigo**: Chamado em `CL_TXS_CONTROLLER.CALCULATE_TAXES` apos HTTP call bem-sucedido; e em `CL_TXS_CONTROLLER.ADD_RESPONSE`
  Fonte: `cl_txs_controller.abap:148-154` e `cl_txs_controller.abap:206-210`
- **O que verificar**:
  - `i_request`, `i_response`, `i_pricing_context` — todos devem estar preenchidos
  - Se vazios: excecao `add_response_buffer_error`
    Fonte: `cl_txs_response_buffer.abap:255-264`
  - `lv_request_hash` calculado — se vazio, gera excecao
    Fonte: `cl_txs_response_buffer.abap:271-277`
  - Feature toggle `GLO_LOG_BR_EXT_JSON_RESP` — afeta o caminho de gravacao
