## Visao Geral

A estrutura raiz `TXS_PAYLOAD` contem os campos de header (nivel de documento) mais referencias a sub-estruturas de tabelas internas (`_items`, `_locations`, `_party`, `_bp_exemption_details`, `_extensions`). Este documento foca exclusivamente nos **campos escalares do header** -- como sao populados, de onde vem os dados e onde fazer debug.

A classe base `CL_TXS_PAYLOAD` implementa a interface `IF_TXS_PAYLOAD` e constroi o payload completo. Para o Brasil, a classe `CL_TXS_PAYLOAD_BR` herda de `CL_TXS_PAYLOAD` e redefine metodos especificos: `BUILD_HEADER`, `GET_PAYLOAD_DATE`, `GET_GROSS_OR_NET`, `GET_OPERATION_NATURE_CODE` e `IS_GROSS_PRICE_ACTIVE`.

**Fonte**: cl_txs_payload.abap:1-61, cl_txs_payload_br.abap:1-45

## Sequencia de Build

O metodo `IF_TXS_PAYLOAD~BUILD` em `CL_TXS_PAYLOAD` (cl_txs_payload.abap:219-227) executa a construcao do payload na seguinte ordem:

```
1. INITIALIZE          â†’ CLEAR payload (limpa todos os campos)
2. BUILD_HEADER        â†’ Popula campos escalares do header (id, date, saleor_purchase, gross_or_net, currency, company_id)
                          BR: chama super->build_header() + adiciona operation_nature_code
3. BUILD_ITEM          â†’ Constroi tabela _items via CL_TXS_ITEMS=>GET_INSTANCE
4. BUILD_LOCATIONS     â†’ Constroi tabela _locations via CL_TXS_LOCATIONS=>GET_INSTANCE
5. BUILD_PARTY         â†’ Constroi tabela _party via CL_TXS_PARTY=>GET_INSTANCE
6. BUILD_BP_EXEMPTION  â†’ Constroi tabela _bp_exemption_details via CL_TXS_BP_EXEMPTIONS=>GET_INSTANCE
7. APPLY_CUSTOM_LOGIC  â†’ Executa BAdI BD_GSLOG_EXT_TAX_CALC_PAYLOAD::EXTEND_PAYLOAD
                          (pode modificar QUALQUER campo do payload, incluindo header e items)
```

**Fonte**: cl_txs_payload.abap:219-227

### Build Header -- Classe Base vs. Brasil

A classe base `CL_TXS_PAYLOAD::BUILD_HEADER` (cl_txs_payload.abap:103-111) popula:


A classe Brasil `CL_TXS_PAYLOAD_BR::BUILD_HEADER` (cl_txs_payload_br.abap:52-57) estende:

```abap

```

## Logica Detalhada dos Campos Header

### Campo: `id`

- **Classe**: `CL_TXS_PAYLOAD::BUILD_HEADER` (cl_txs_payload.abap:104)
- **Metodo de geracao**: `dao->generate_payload_id()` que chama `CL_SYSTEM_UUID=>CREATE_UUID_X16_STATIC()` (cl_txs_dao.abap:120-124)
- **Tipo**: UUID hexadecimal de 16 bytes (32 caracteres)
- **Observacao**: Gerado a cada chamada do TXS. Nao reutilizado entre chamadas.

---

### Campo: `company_id`

- **Classe**: `CL_TXS_PAYLOAD::BUILD_HEADER` (cl_txs_payload.abap:109)
- **Fonte SAP**: `pricing_context->get_header( )-bukrs` que mapeia para `KOMK-BUKRS`
- **Significado**: Codigo da empresa (company code) do documento
- **Valor no pricing context**: `cl_txs_pricing_context::get_currency` le de `pricing_header-waerk` (cl_txs_pricing_context.abap:190-191), e analogamente `company_id` vem de `pricing_header-bukrs`

---

### Campo: `date`

- **Classe**: `CL_TXS_PAYLOAD_BR::GET_PAYLOAD_DATE` (redefined) -- cl_txs_payload_br.abap:257-325
- **Formato**: ISO 8601 com precisao de milissegundos: `YYYY-MM-DDTHH:MM:SS.000Z`
- **Fuso horario**: Converte para o timezone da planta via `CL_TXS_FM_DATE_TIME=>GET_INSTANCE( )->GET_LOCATION_TIMEZONE` usando pais (`land1`), regiao (`regio`) e CEP (`pstlz`) da planta. Se nao encontrar, usa `SY-ZONLO`.

**Arvore de decisao (Brasil)**:

```
1. Aplicacao = SD (KAPPL = 'V')?
   |
   +-- E billing doc transfer (document_type in accounting_doc_type-billing_doc_transfer)?
   |     â†’ date = KOMK-FKDAT (billing date)
   |
   +-- E stock transfer (STO)?
   |     â†’ KOMK-FKDAT se preenchido, senao KOMK-ERDAT (creation date)
   |
   +-- Outros SD
         â†’ date = KOMK-PRSDT (pricing date)

2. Stock transfer para MM (STO via MM, nao SD)?
   â†’ date = KOMK-PRSDT (pricing date)

3. Default (outros MM)
   â†’ super->get_payload_date()
   â†’ pricing_context->get_date() (= SY-DATUM) + pricing_context->get_local_time() (= SY-TIMLO)
   â†’ Formato: YYYY-MM-DDTHH:MM:SS.000Z no timezone do sistema
```

**Detalhe de conversao de timezone** (cl_txs_payload_br.abap:312-322):

Nos cenarios 1 e 2, apos determinar `lv_new_date`, o codigo converte o horario do sistema (`SY-TIMLO`) para o timezone da planta:

```abap
CONVERT DATE lv_new_date TIME sy-timlo INTO TIME STAMP lv_timestamp TIME ZONE sy-zonlo.
CONVERT TIME STAMP lv_timestamp TIME ZONE lv_plant_timezone INTO DATE lv_date_plant TIME lv_time_plant.
```

A data final usa `lv_new_date` (nao `lv_date_plant`) porque apenas o horario deve ser ajustado, nao a data.

**Fonte**: cl_txs_payload_br.abap:257-325, cl_txs_payload.abap:185-196

### Campo: `saleor_purchase`

- **Classe**: `CL_TXS_PAYLOAD::GET_SALES_OR_PURCHASE` (cl_txs_payload.abap:199-211)
- **Nao redefinido** pelo Brasil -- usa a implementacao da classe base

**Logica**:

```
pricing_context->get_application_for_payload() retorna KAPPL:
  |
  +-- 'V' (application-sales_and_distribution) â†’ internal_code API_SALE
  |     â†’ cl_txs_internal_codes=>api_sale_or_purchase-sale
  |     â†’ Valor: conforme tabela de internal codes (tipicamente 'sale')
  |
  +-- 'TX' (application-taxes) â†’ internal_code API_PURCHASE
        â†’ cl_txs_internal_codes=>api_sale_or_purchase-purchase
        â†’ Valor: conforme tabela de internal codes (tipicamente 'purchase')
```

**Constantes de referencia**:
- `if_txs_constants=>application-sales_and_distribution` = `'V'` (Fonte: if_txs_constants.abap:8)
- `if_txs_constants=>application-taxes` = `'TX'` (Fonte: if_txs_constants.abap:7)
- `cl_txs_internal_codes=>api_sale_or_purchase-sale` = `'API_SALE'` (Fonte: cl_txs_internal_codes.abap:200)
- `cl_txs_internal_codes=>api_sale_or_purchase-purchase` = `'API_PURCHASE'` (Fonte: cl_txs_internal_codes.abap:201)

**Observacao**: O valor final enviado na API depende da tabela de internal codes (`TXS_INT_CODES`), onde o internal code `API_SALE` e mapeado para um valor textual (ex: `'sale'`). O metodo `get_single_internal_code_value()` faz essa resolucao.

### Campo: `gross_or_net` (Redefinicao BR)

- **Classe**: `CL_TXS_PAYLOAD_BR::GET_GROSS_OR_NET` (cl_txs_payload_br.abap:76-98)
- **Redefinido no Brasil** -- a classe base simplesmente retorna `'net'` via internal codes

**Arvore de decisao (Brasil)**:

```
1. E transacao de downpayment? (pricing_context->is_downpayment_transaction())
   OU e accounts receivable? (logbr_down_payt_scen_valdtr->is_accounts_receivable())
   OU e customer down payment? (logbr_down_payt_scen_valdtr->is_customer_down_payment())
   |
   +-- SIM â†’ 'gross' (via internal_code API_GROSS)
   |
   +-- NAO â†’ continua...

2. Gross price ativo? (is_gross_price_active())
   E aplicacao = TX (MM)? (application-taxes)
   E NAO e invoice? (is_invoice_transaction() = false)
   |
   +-- SIM (todas as 3 condicoes) â†’ 'gross' (via internal_code API_GROSS)
   |
   +-- NAO â†’ 'net' (via internal_code API_NET)
```

**Metodo `IS_GROSS_PRICE_ACTIVE`** (cl_txs_payload_br.abap:357-366):
- Obtem o tipo de material via `pricing_context->get_material_type()`
- Converte para product type via `CL_TXS_LOGBR_MAT_TYPES_DA_ADPT`
- Verifica se gross price esta ativo para o par (product_type, tax_code) via `CL_TXS_LOGBR_GRS_PRC_ACT_ADPTR=>GET_INSTANCE( )->IS_ACTIVE_BY_ITEM_LINE`

**Constantes de referencia**:
- `cl_txs_internal_codes=>api_gross_or_net-gross` = `'API_GROSS'` (Fonte: cl_txs_internal_codes.abap:205)
- `cl_txs_internal_codes=>api_gross_or_net-net` = `'API_NET'` (Fonte: cl_txs_internal_codes.abap:206)
- `if_txs_constants=>application-taxes` = `'TX'` (Fonte: if_txs_constants.abap:7)

**Classe base** (cl_txs_payload.abap:161-168): Sempre retorna `'net'` via internal code `API_NET`, sem nenhuma logica condicional.

## Instanciacao via Factory

O `CL_TXS_PAYLOAD_FACTORY` (cl_txs_payload_factory.abap:51-79) determina qual classe de payload instanciar:

1. Determina o tipo de payload via `CL_TXS_PAYLOAD_TYPE=>GET_INSTANCE`
2. Tenta instanciar `{payload_type}_{country}` (ex: `CL_TXS_PAYLOAD_BR`)
3. Se a classe especifica do pais nao existir (`CX_SY_DYN_CALL_ILLEGAL_CLASS`), instancia `{payload_type}` (classe base)

Este e o padrao **Country Specialization** usado em todo o TXS: tenta primeiro a classe com sufixo `_BR`, e faz fallback para a classe base.

**Fonte**: cl_txs_payload_factory.abap:51-79

## Extensibilidade

### BAdI: BD_GSLOG_EXT_TAX_CALC_PAYLOAD

- **Metodo**: `EXTEND_PAYLOAD`
- **Chamado em**: `CL_TXS_PAYLOAD::APPLY_CUSTOM_LOGIC` (cl_txs_payload.abap:68-92)
- **Momento**: APOS todos os builds (header, item, locations, party, bp_exemption)
- **Pode modificar**: **QUALQUER** campo do payload (header e sub-estruturas), pois recebe `CHANGING payload = payload`

**Parametros exportados pelo BAdI**:

| Parametro | Tipo | Descricao |
|-----------|------|-----------|
| `pricing_header` | Estrutura KOMK | Header do pricing |
| `pricing_item` | Estrutura KOMP | Item do pricing |
| `lfd_for_consumption` | `IF_GSLOG_LFD_FOR_CONSUMPTION` | Local Fiscal Data para consumo |

**Parametro CHANGING**:

| Parametro | Tipo | Descricao |
|-----------|------|-----------|
| `payload` | `TXS_PAYLOAD` | Payload completo (header + sub-estruturas) |

**Fonte**: cl_txs_payload.abap:68-92

### Campos tipicamente preenchidos via BAdI

Os seguintes campos do header **nao sao populados no standard** e so podem ser preenchidos via este BAdI:

- `cash_discount_percent` -- Percentual de desconto financeiro
- `business_partner_id` -- ID do parceiro de negocios (fornecedor/cliente)
- `is_transaction_in_tax_group` -- Indicador de grupo fiscal
- `is_comp_deferred_tax_enabled` -- Indicador de imposto diferido
- `is_cash_discount_planned` -- Indicador de desconto financeiro planejado

### Extensao da Estrutura DDIC

A estrutura `TXS_PAYLOAD` possui annotation `@AbapCatalog.enhancement.category : #EXTENSIBLE_ANY`, permitindo adicionar novos campos via append structure. Campos adicionados podem ser populados via BAdI `EXTEND_PAYLOAD`.
