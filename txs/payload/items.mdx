## Visao Geral

O payload TXS envia ao motor externo de calculo uma lista de **items** (estrutura `TXS_ITEM`). Cada item contem quatro sub-estruturas:

1. **item\_classifications** (`TXS_ITEM_CLASSIFICATION_TTY`) -- Classificacoes fiscais (NCM, CEST, EAN, servicos)

2. **additional\_item\_information** (`TXS_ADDIT_ITEM_INFORMATION_TTY`) -- Informacoes adicionais (origem, uso, producao propria)

3. **cost\_information** (`TXS_COST_INFORMATION_TTY`) -- Custos acessorios (frete, seguro, desconto)

4. **exemption\_details** (`TXS_EXEMPTION_DETAIL_TTY`) -- Detalhes de isencao

O **fluxo de montagem** (Fonte: cl\_txs\_items.abap:84-91) e:

```
CL_TXS_ITEMS::ADD_ITEM
  -> build_header()              --> campos diretos do item
  -> build_item_classifications() --> CL_TXS_ITEM_CLASSIFICATIONS_BR
  -> build_additional_item_info() --> CL_TXS_ADDITIONAL_ITEM_INFO_BR
  -> build_cost_information()     --> CL_TXS_COST_INFORMATION_BR
  -> build_exemption_detail()     --> CL_TXS_MAT_EXEMPTIONS
```

A classe BR herda da classe base via **Country Specialization**: `CL_TXS_ITEMS_BR` herda `CL_TXS_ITEMS`, e a factory `CL_TXS_FACTORY` instancia a variante BR quando `country = 'BR'`.

### 1.3 Logica quantity BR

Fonte: cl\_txs\_items\_br.abap:55-78

A classe BR sobrescreve `GET_QUANTITY` com logica especifica:

```
SE feature toggle GLO_LOG_BR_MAA_NF_AFD_MAPPING esta ativo:
  -> GET_QUANTITY_TAXABLE()
     -> pricing_context->get_item_quantity_taxable()
     -> SE resultado INITIAL
        E documento e invoice (cl_txs_business_transaction=>is_invoice)
        E PO tem multi-account assignment (has_mul_acct_assgm_dist_amount):
          -> fallback para KOMP-MGAME
SENAO:
  -> super->GET_QUANTITY()  (classe base: pricing_context->get_item_quantity_taxable)
```

**Feature toggle**: `GLO_LOG_BR_MAA_NF_AFD_MAPPING` (Fonte: cl\_txs\_items\_br.abap:57)

**Fallback MGAME**: Quando a invoice tem distribuicao de conta multipla, a quantidade do item pricing pode estar zerada. Nesse caso, utiliza-se `KOMP-MGAME` como fallback.

### 1.4 Logica unit\_price -- Despacho

Fonte: cl\_txs\_items\_br.abap:93-103

O metodo `GET_UNIT_PRICE` da classe BR chama primeiro `super->get_unit_price()` (classe base) e depois despacha conforme application:

| Application  | Constante                                              | Valor  | Metodo BR           |
| ------------ | ------------------------------------------------------ | ------ | ------------------- |
| MM (Compras) | `if_txs_constants=>application-taxes`                  | `'TX'` | `GET_UNIT_PRICE_MM` |
| SD (Vendas)  | `if_txs_constants=>application-sales_and_distribution` | `'V'`  | `GET_UNIT_PRICE_SD` |

Fonte das constantes: if\_txs\_constants.abap:7-8

#### 1.4.1 Logica unit\_price MM

Fonte: cl\_txs\_items\_br.abap:107-145

```
1. Determinar quantidade:
   - SE feature toggle GLO_LOG_BR_MAA_NF_AFD_MAPPING ativo:
       lv_quantity = GET_QUANTITY_TAXABLE()
   - SENAO:
       lv_quantity = pricing_context->get_item_quantity_taxable()

2. Determinar condition type para base amount:
   - Internal code: cl_txs_internal_codes=>condition_type-item_base_amount
     = 'CONDITION_TYPE_BASE_AMOUNT'  (Fonte: cl_txs_internal_codes.abap:32)
   - SE feature toggle GLO_LOG_BR_BKJ_ROUND ativo:
       component = cl_txs_internal_codes=>integration_component-mm = 'MM'
     SENAO:
       component = pricing_context->get_component()
   - Consulta tabela de internal codes para obter o condition type KSCHL

3. Buscar KWERT do pricing:
   - pricing_context->get_condition_record_by_type(condition_type)

4. Calcular:
   - SE KWERT e quantidade nao-initial:
       unit_price = KWERT / quantidade
   - SENAO:
       unit_price = net_price (fallback da classe base)

5. Formatacao: 12 casas decimais, separador ponto, sem expoente
```

**Feature toggle**: `GLO_LOG_BR_BKJ_ROUND` (Fonte: cl\_txs\_items\_br.abap:19, valor `'GLO_LOG_BR_BKJ_ROUND'`)

Quando este toggle esta ativo, o componente fixo `'MM'` e usado na busca do internal code em vez do componente dinamico do pricing context. Isso garante que a condition type correta de MM seja usada mesmo em cenarios especiais.

#### 1.4.2 Logica unit\_price SD

Fonte: cl\_txs\_items\_br.abap:148-166

```
1. Determinar condition type:
   - GET_COND_TYPE_UNIT_PRICE_BASE()
     -> dao->select_cond_type_by_int_code(
          i_code = if_txs_constants_br=>internal_code-taas_unit_price_base
          i_application = pricing_context->get_application_of_pricing()
        )
   - Internal code: 'TAAS_UNIT_PRICE_BASE'
     (Fonte: if_txs_constants_br.abap:45)

2. Buscar KWERT do pricing:
   - pricing_context->get_condition_record_by_type(condition_type)

3. Calcular:
   - SE KWERT e quantidade nao-initial:
       unit_price = KWERT / quantidade
   - SENAO:
       unit_price = net_price (fallback)

4. Formatacao: 12 casas decimais, separador ponto, sem expoente
```

### 2.5 Classificacoes para Servico

Fonte: cl\_txs\_item\_classifications\_br.abap:144-228

Despacha entre MM e SD conforme application:

| Application | Metodo                         | Classificacoes                               |
| ----------- | ------------------------------ | -------------------------------------------- |
| MM (Taxes)  | `APPEND_ITEM_CLASS_SERVICE_MM` | servTypeExec, servTypeFed, servTypeFrom, NBS |
| SD (Sales)  | `APPEND_ITEM_CLASS_SERVICE_SD` | servTypeExec, servTypeFed, servTypeFrom, NBS |

#### Detalhamento por System Code (Servico)

| System Code      | Metodo MM                | Metodo SD                | Fonte do Valor                                                                                                                              |
| ---------------- | ------------------------ | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------- |
| **servTypeExec** | `GET_MM_SERVICE_EXEC`    | `GET_SD_SERVICE_EXEC`    | `dao_br->get_service_type(country, material, jurisdiction_code)` -- codigo do tipo de servico municipal (ABRASF) no local de execucao       |
| **servTypeFed**  | `GET_MM_SERVICE_FEDERAL` | `GET_SD_SERVICE_FEDERAL` | `dao_br->get_abrasf_code(appl, country, material, jurisdiction_code)` -- codigo ABRASF federal, formatado via `FORMAT_ABRASF_CODE()`        |
| **servTypeFrom** | `GET_MM_SERVICE_FROM`    | `GET_SD_SERVICE_FROM`    | `dao_br->get_service_type(country, material, jurisdiction_code)` -- codigo do tipo de servico no local do fornecedor (MM) ou da planta (SD) |
| **NBS**          | `GET_NBS`                | `GET_NBS`                | `logbr_nbs_reader_da->read_nbs_by_material(MATNR)` -- Nomenclatura Brasileira de Servicos                                                   |

**servTypeExec MM** (Fonte: cl\_txs\_item\_classifications\_br.abap:413-442):

- Usa endereco de entrega (delivery address) via `CL_TXS_MM_ADDRESS_ADAPTER` para obter `TXJCD`

- Fallback para `pricing_context->get_plants_or_branches()-txjcd`

- Resultado: `servtype_in` (tipo de servico de entrada)

**servTypeExec SD** (Fonte: cl\_txs\_item\_classifications\_br.abap:505-523):

- Usa jurisdiction code do **cliente** (`dao_br->select_customer_by_number`)

- Resultado: `servtype_out` (tipo de servico de saida)

**servTypeFrom MM** (Fonte: cl\_txs\_item\_classifications\_br.abap:470-488):

- Usa jurisdiction code do **fornecedor** (`dao->select_supplier_by_number`)

- Resultado: `servtype_in`

**servTypeFrom SD** (Fonte: cl\_txs\_item\_classifications\_br.abap:551-569):

- Usa jurisdiction code da **planta** (`get_plants_or_branches()-txjcd`)

- Resultado: `servtype_out`

### 2.6 Classificacoes para STO (Stock Transfer Order)

Fonte: cl\_txs\_item\_classifications\_br.abap:156-178

Metodos STO-especificos: `GET_MATERIAL_NCM_STO`, `GET_MATERIAL_CEST_STO`, `GET_MATERIAL_EAN_STO`, `GET_MATERIAL_EXTIPI_STO`

| System Code | Metodo STO                | Diferenca do Fluxo Normal                                                                                                                                |
| ----------- | ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ncm**     | `GET_MATERIAL_NCM_STO`    | Usa `READ_MATERIAL_NCM_STO()` que determina a planta (supplying ou receiving) conforme direcao do STO e le o NCM via `dao->read_material_control_code()` |
| **cest**    | `GET_MATERIAL_CEST_STO`   | Usa `dao_br->determine_cest()` com NCM formatado (com pontos)                                                                                            |
| **ean**     | `GET_MATERIAL_EAN_STO`    | Le EAN do `EKPO` via `dao->get_po_item_by_num()` usando PO number/item do STO                                                                            |
| **exTipi**  | `GET_MATERIAL_EXTIPI_STO` | Extrai ultimos 2 digitos (NCM 10 chars) ou 3 digitos (NCM > 10 chars) do NCM STO                                                                         |

**Logica READ\_MATERIAL\_NCM\_STO** (Fonte: cl\_txs\_item\_classifications\_br.abap:572-611):

```
SE STO outgoing, GI return without GR, ou scrap:
  planta = supplying_plant
SE STO incoming:
  planta = receiving_plant

NCM = dao->read_material_control_code(material, planta)
Remove pontos do NCM
```
