## O que e ABAP Cloud

ABAP Cloud e o modelo de desenvolvimento para S/4HANA Cloud. Utiliza um subconjunto restrito do ABAP, garantindo compatibilidade com upgrades automaticos da SAP.

## Regras Fundamentais

### 1. Apenas Released APIs
- Todo acesso a objetos SAP deve ser via APIs com Release State **C1** (Released)
- Verificar no ADT: Properties → API State → deve ser "Released" (C1)
- Se a API nao e Released, ela NAO pode ser usada no Public Cloud
- Alternativa: buscar API Released equivalente ou usar extensibilidade Key User

### 2. Clean Core
- Zero modificacoes no codigo standard SAP
- Extensoes apenas via pontos oficiais (BAdIs, Enhancement Spots)
- Custom code isolado do standard
- Compatibilidade com upgrades automaticos garantida

### 3. ABAP Restrito
Proibido no ABAP Cloud:
- `CALL FUNCTION` para FMs nao Released
- `SELECT` direto em tabelas DB nao Released (usar CDS Views Released)
- `EXEC SQL` / Native SQL
- `GENERATE SUBROUTINE POOL`
- `CALL TRANSACTION`
- `SUBMIT`
- `AUTHORITY-CHECK` classico (usar nova API de autorizacao)
- `WRITE` / `SKIP` / Classic List Processing
- `FORM...ENDFORM` (usar metodos)
- `COMMUNICATION` statements
- Acesso a cluster/pool tables

### 4. Substituicoes Comuns

| ABAP Classico | ABAP Cloud |
|---------------|------------|
| `CALL FUNCTION 'xxx'` | Classe Released equivalente |
| `SELECT FROM <db_table>` | `SELECT FROM <cds_view>` (Released) |
| `AUTHORITY-CHECK` | `cl_abap_authorization=>check_authorization()` |
| `CALL TRANSACTION` | Navigate via Fiori Launchpad / API |
| `SUBMIT report` | Classe Released / Job Scheduling API |
| `READ TABLE ... BINARY SEARCH` | `READ TABLE ... sorted/hashed` |
| `FORM...ENDFORM` | `METHOD...ENDMETHOD` |
| `WRITE` statement | RAP + Fiori UI |
| `MESSAGE` (classic) | `cx_*` exception classes / RAP messages |

## Tiers de Extensibilidade

### Tier 1 - ABAP Cloud (Developer Extensibility)
- Ambiente: ABAP Environment (Steampunk) ou Embedded Steampunk
- Linguagem: ABAP Cloud (subconjunto restrito)
- APIs: Apenas Released (C1)
- Ferramentas: ADT (Eclipse), BAS (Business Application Studio)
- Artefatos: CDS Views, RAP BOs, Classes, Interfaces

### Tier 2 - Key User Extensibility
- Ambiente: S/4HANA Cloud diretamente
- Ferramentas: Fiori Apps de extensibilidade
- Artefatos: Custom Fields, Custom Logic, Custom CDS Views, Custom BOs
- Sem codigo (ou ABAP muito restrito)

### Tier 3 - Classic ABAP (NAO disponivel no Public Cloud)
- Apenas On-Premise e Private Cloud
- ABAP completo sem restricoes
- NAO compativel com upgrades automaticos

## Verificacao de Released APIs

### No ADT (Eclipse)
1. Abrir o objeto (classe, CDS View, tabela)
2. Properties → API State
3. Deve mostrar: **Released** (C1) com "Use System-Internally (C1)"

### Programaticamente
```abap
" Verificar se uma API e Released
SELECT FROM I_CustABAPObjDirectoryEntry
  FIELDS ABAPObject, ReleaseState
  WHERE ABAPObject = 'CL_EXAMPLE'
  INTO TABLE @DATA(lt_result).
" ReleaseState = 'RELEASED' → pode usar
```

## Boas Praticas ABAP Cloud

1. **Sempre usar CDS Views** em vez de SELECT direto em tabelas
2. **RAP para CRUD** em vez de INSERT/UPDATE/DELETE direto
3. **Exception Classes** em vez de MESSAGE/SY-SUBRC
4. **Inline Declarations** (`DATA(...)`, `FIELD-SYMBOL(...)`)
5. **NEW/VALUE/CONV/COND/SWITCH** construtores
6. **String Templates** (`|text { var }|`) em vez de CONCATENATE
7. **ABAP Unit** para todos os artefatos custom
8. **Naming Conventions**: ZCL_ (class), ZI_ (interface), Z*_R_ (CDS Root), Z*_C_ (CDS Projection)
